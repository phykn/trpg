# Browser QA Skill

이 스킬은 Codex가 열린 브라우저의 실제 게임 UI를 조작하며 개선사항을 찾기 위한 절차다. 목적은 자동 회귀 테스트가 아니라 탐색 QA다. API를 직접 때려서 진행하지 말고, 플레이어가 쓰는 화면 입력, 버튼, 모달, 판정 UI를 사용한다.

## 입력

- 열린 브라우저 탭의 TRPG 게임 화면
- 선택 입력: 플레이할 프로필, 목표 턴 수, 집중 영역

기본값:

- 목표 턴 수: 10턴
- 집중 영역: 전체 플레이 가능성
- 저장 위치: `qa_test/browser/<YYYYMMDD-HHMMSS>/report.md`

## 사전 점검

1. 브라우저 스킬로 현재 탭을 확인한다. 브라우저 관찰을 사용할 수 없으면 자동 runner나 직접 API 호출로 대체하지 말고 중단해 사용자에게 보고한다.
2. 게임 화면이 열려 있지 않으면 사용자에게 URL 또는 실행 여부를 한 번만 물어본다.
3. 서버나 클라이언트가 꺼져 있으면 직접 추측하지 말고 보이는 에러와 필요한 실행 명령을 보고한다.
4. 콘솔 에러가 있으면 시작 상태에 기록한다.
5. QA는 이 스킬로만 진행한다. 별도 자동 runner나 직접 API 호출로 대체하지 않는다.

## 판정 기준

플레이 중 아래 기준으로 개선사항을 찾는다.

### Gameplay

- `ExposedTransitionValidity`: 중요한 행동이 가능하거나 막힌 이유를 플레이어가 알 수 있는가?
- `FictionContinuity`: NPC, 단서, 위치, 퀘스트, 이전 선택이 이어지는가?
- `AgencyContinuity`: 플레이어 의도가 왜곡되지 않고 다음 시도를 상상할 수 있는가?

### UI

- `Situation`: 다음 결정을 바꾸는 상황 정보가 보이는가?
- `ActionSpace`: 가능한 접근 범주가 보이되 숨은 정답을 누설하지 않는가?
- `Feedback`: 입력이 처리되었고 무엇이 달라졌는지 알 수 있는가?
- `Continuity`: 미해결 목표, 제약, 관계, 자원, 위험을 다시 찾을 수 있는가?

비밀 정보가 UI나 나레이션에 먼저 드러나면 이슈다. 공개 단서는 관찰 가능한 징후, `traits`, `background`, 공개 `knowledge`에 있어야 한다.

## 플레이 절차

1. 화면을 관찰하고 현재 상태를 짧게 기록한다.
2. 한 턴마다 실제 플레이어처럼 하나의 한국어 행동을 입력한다.
3. 확인 모달이나 판정 UI가 나오면 화면에 보이는 흐름대로 처리한다.
4. 매 턴 후 다음을 기록한다.
   - 입력
   - 관찰된 GM 반응 또는 UI 반응
   - 바뀐 상태
   - 헷갈린 점 또는 막힌 점
   - 콘솔 에러나 시각적 문제
5. 같은 문제가 반복되면 더 파고들기보다 재현 절차를 안정화한다.
6. 치명적인 진행 불능, 저장 손상, 화면 전체 오류가 나오면 중단하고 그 지점을 리포트한다.

## 플레이 스타일

- 일반 플레이어가 할 법한 행동을 한다.
- 한 턴에 여러 의도를 섞지 않는다.
- UI에 보이지 않는 내부 정보를 사용하지 않는다.
- 숨은 정답을 일부러 맞히려 하지 않는다.
- 한국어 2인칭 존댓말, 기술 용어, 퀘스트/전투/판정 흐름을 유심히 본다.
- 화면상 선택지가 있더라도 자유 입력이 가능한지 주기적으로 확인한다.

## 이슈 분류

각 이슈는 아래 중 하나 이상으로 분류한다.

- `server`: 런타임 규칙, 나레이션, 판정, 저장, API 응답 문제
- `client`: UI 표시, 입력, 모달, 상태 반영, 콘솔 에러 문제
- `scenario`: seed 내용, 단서, NPC, 퀘스트, 공개/비공개 정보 문제
- `ux`: 플레이어 이해, 다음 행동 추론, 피드백, 복귀성 문제
- `unknown`: 표면만으로 원인을 좁히기 어려운 문제

심각도:

- `P0`: 진행 불능, 데이터 손상, 화면 전체 오류
- `P1`: 핵심 플레이 루프가 깨지거나 잘못된 결정을 유도
- `P2`: 반복 혼란, 중요한 정보 누락, 불편한 우회 필요
- `P3`: 문구, 작은 시각 문제, 낮은 빈도 개선점

## 금지

- 브라우저 QA 중 발견한 문제를 사용자 요청 없이 바로 수정하지 않는다.
- 내부 API 호출만으로 진행하고 UI 관찰을 생략하지 않는다.
- 화면에 없는 숨은 상태를 확정 사실처럼 쓰지 않는다.
- 턴 번호 없는 이슈를 만들지 않는다.
- 브라우저 관찰이 아닌 추측을 이슈 근거로 쓰지 않는다.
