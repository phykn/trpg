# Browser QA Skill

이 스킬은 Codex가 열린 브라우저의 실제 게임 UI를 조작하며 개선사항을 찾기 위한 절차다. 목적은 자동 회귀 테스트가 아니라 탐색 QA다. API를 직접 때려서 진행하지 말고, 플레이어가 쓰는 화면 입력, 버튼, 모달, 판정 UI를 사용한다.

## 입력

- 열린 브라우저 탭의 TRPG 게임 화면
- 선택 입력: 플레이할 프로필, 목표 턴 수, 집중 영역

기본값:

- 목표 턴 수: 10턴
- 집중 영역: 전체 플레이 가능성
- 저장 위치: `qa_test/browser/<YYYYMMDD-HHMMSS>/report.md`

모드:

- `exploratory`: 기본 10턴 탐색. 개선점 발굴이 목적.
- `ending`: 사용자가 엔딩까지 요청한 경우. 턴 수보다 엔딩 도달/진행 불능 지점이 기준.
- `regression`: 특정 수정사항 확인. 요청받은 항목을 먼저 검증하고, 주변 회귀를 짧게 확인.

## 사전 점검

1. 브라우저 스킬로 현재 탭을 확인한다. 브라우저 관찰을 사용할 수 없으면 자동 runner나 직접 API 호출로 대체하지 말고 중단해 사용자에게 보고한다.
2. 게임 화면이 열려 있지 않으면 사용자에게 URL 또는 실행 여부를 한 번만 물어본다.
3. 서버나 클라이언트가 꺼져 있으면 직접 추측하지 말고 보이는 에러와 필요한 실행 명령을 보고한다. 사용자가 서버 실행까지 맡긴 경우에만 직접 실행한다.
4. 콘솔 에러가 있으면 시작 상태에 기록한다.
5. QA는 이 스킬로만 진행한다. 별도 자동 runner나 직접 API 호출로 대체하지 않는다.
6. 새 시나리오나 수정된 시나리오를 QA하는 경우, 가능하면 브라우저 시작 전에 `agency.story.tool sweep`과 `runtime-smoke` 결과를 확인한다. 이 검사는 브라우저 QA를 대체하지 않는다.

## 브라우저 입력 트러블슈팅

`Browser Use virtual clipboard is not installed` 오류가 나면 앱 입력창 문제가 아니라 in-app browser 자동화 세션의 가상 클립보드 상태가 꼬인 것이다. `fill`/`type`은 한글 입력을 위해 페이지에 임시 `navigator.clipboard`를 심고 paste 이벤트로 입력하는데, 자동화 런타임의 설치 캐시와 실제 페이지 상태가 어긋나면 이 오류가 난다.

복구 절차:

1. 같은 locator를 반복 재시도하지 않는다.
2. 브라우저 자동화 런타임을 리셋한다.
3. in-app browser에 다시 연결하고 새 탭을 연다.
4. `http://localhost:8081/`로 이동한다.
5. 입력창에 짧은 한글 문장을 `fill`해서 성공 여부를 확인한 뒤 QA를 재개한다.

Node REPL 도구를 쓰는 경우에는 `mcp__node_repl__js_reset` 도구를 호출한다. 리셋 후 브라우저 스킬의 첫 연결 셀을 다시 실행하고, 새 탭에서 `fill`을 검증한다. 이 오류만으로 서버나 클라이언트를 수정하지 않는다.

## 판정 기준

플레이 중 아래 기준으로 개선사항을 찾는다.

### Gameplay

- `ExposedTransitionValidity`: 중요한 행동이 가능하거나 막힌 이유를 플레이어가 알 수 있는가?
- `FictionContinuity`: NPC, 단서, 위치, 퀘스트, 이전 선택이 이어지는가?
- `AgencyContinuity`: 플레이어 의도가 왜곡되지 않고 다음 시도를 상상할 수 있는가?
- `Pace`: 게임이 호쾌하게 진행되는가? 행동 결과가 분명히 나고, 불필요한 설명·확인·반복 판정에 묶이지 않고 다음 장면이나 선택으로 밀고 나가는가?
- `SceneStateBlend`: 나레이션이 담백한 플레이 로그체로 읽히며, 장면 약 70%와 상태 정보 약 30%가 자연스럽게 섞이는가?

### UI

- `Situation`: 다음 결정을 바꾸는 상황 정보가 보이는가?
- `ActionSpace`: 가능한 접근 범주가 보이되 숨은 정답을 누설하지 않는가?
- `Feedback`: 입력이 처리되었고 무엇이 달라졌는지 알 수 있는가?
- `Continuity`: 미해결 목표, 제약, 관계, 자원, 위험을 다시 찾을 수 있는가?

비밀 정보가 UI나 나레이션에 먼저 드러나면 이슈다. 공개 단서는 관찰 가능한 징후, `traits`, `background`, 공개 `knowledge`에 있어야 한다.

진행감은 빠른 스킵이 아니라 선명한 처리다. 플레이어가 행동하면 성공/실패, 상태 변화, 다음 선택지가 또렷해야 하고, 같은 정보를 다시 확인시키거나 세계관 설명으로 턴을 늘리면 이슈다. 중요한 결정은 확인하되, 일반 진행은 과감하게 다음 장면으로 넘긴다.

로그 품질은 담백한 플레이 로그체 기준으로 본다. `성공했습니다`, `피해를 입었습니다`처럼 상태만 보고하면 이슈고, `묘한 긴장감`, `알 수 없는 기척`처럼 추상어만 남기고 결과가 안 보이면 이슈다. 좋은 로그는 움직임, 충돌/반응, 현재 상태가 짧게 이어지고, 성공/실패·대상 변화·위치·자세·감정·전투 가능 여부 중 이번 턴에 중요한 상태가 장면 안에서 드러난다. 화려한 비유, 장중한 표현, 감정 과잉은 이슈다.

### Tutorial / Scenario

- `InfoFirst`: 튜토리얼 질문에는 장식보다 필요한 정보가 먼저 나오는가? 기준은 정보 70%, 장면 재료 30%다.
- `ConcreteRule`: 세계 규칙은 은유가 아니라 플레이어가 행동에 쓸 수 있는 말로 설명되는가?
- `PlayerKnowledgeFairness`: QA 입력이 플레이어가 화면에서 알 수 있는 정보만 전제로 삼는가?
- `PlaceIdentity`: 플레이어가 현재 위치를 물으면 챕터명이나 주변 세계가 아니라 현재 장소명이 먼저 답으로 나오는가?
- `RequiredProgress`: 필수 튜토리얼 진행이 운, 숨은 정답, 특정 표현 하나에 막히지 않는가?
- `Recoverability`: 실패했을 때 왜 막혔는지와 다음 시도가 보이는가?
- `NamingClarity`: 챕터명, 섬 이름, 실제 장소 이름, 출발지와 도착지가 헷갈리지 않는가?
- `NoPrematureState`: 나레이션이 아직 일어나지 않은 행동을 이미 한 것처럼 쓰지 않는가?
- `ExposedDescriptionTone`: 이동 로그에 직접 노출되는 장소 설명이 나레이션 본문과 같은 존댓말 톤으로 읽히는가?
- `StagingPlace`: 선착장, 문 앞, 입구처럼 다음 이동을 준비하는 장소를 다음 목적지 도착처럼 쓰지 않는가?
- `CurrentActor`: 직전 대화 상대를 현재 장소에 없는 인물처럼 계속 행동시키지 않는가?
- `HandoffLeak`: 다음 사건으로 넘기는 문장이 이전 퀘스트 물건이나 보상 소유를 근거 없이 다시 꺼내지 않는가?
- `ConnectionPriority`: 플레이어가 다음 섬이나 다음 사건 장소를 명시했을 때, 가까운 되돌아가기 연결이 먼저 잡혀 엉뚱한 중간 장소로 빠지지 않는가?
- `TargetResponder`: 특정 NPC에게 묻는 입력에서 다른 NPC가 대신 답하거나 플레이어 질문만 반복하고 끝나지 않는가?
- `SpoilerBoundary`: 다음 섬이나 사건의 정보는 기대와 첫 단서까지만 주고 결론을 말하지 않는가?
- `FirstConflictCoverage`: 튜토리얼 다음 첫 갈등 장에서 대화, 조사, 비교, 선택을 충분히 눌러볼 수 있는가?
- `LaterChapterCoverage`: 첫 갈등 장 이후의 섬도 원인, 현재 피해, 양쪽 근거, 장소 단서, 선택 의미를 질문으로 확인할 수 있는가?
- `RewardVisibility`: 선택 보상 아이템이 선택 전 장소에서 보이거나 주울 수 있는 물건처럼 노출되지 않는가?
- `DataPromptBoundary`: catalog나 고정 라벨이 LLM 지시문처럼 쓰이거나, 장면 규칙을 대신 설명하지 않는가?
- `RequiredProgressGate`: 필수 진행이 주사위 성공 하나에 잠겨 있지 않은가? 실패해도 다시 시도하거나, 핵심 장소 진입·관찰·의뢰 수락 같은 확정 행동으로 선택지가 열려야 한다.

튜토리얼은 특히 초보 플레이어가 이것저것 물어본다고 가정한다. 단, QA 입력은 플레이어가 이미 화면에서 알 수 있는 말만 쓴다. 화면에 동행자가 필요하다는 단서가 없으면 `왜 동행자가 필요한지`라고 묻지 말고, 먼저 `배를 탈 수 있는지`, `출항 규칙이 무엇인지`, `저 사람은 누구인지`처럼 자연스럽게 묻는다.
플레이어가 모르는 전제를 물어야만 진행 정보가 나오면 `PlayerKnowledgeFairness` 이슈다. 필수 정보는 시작 장면, 현재 장소, 보이는 NPC, 공개 지식 중 하나에서 먼저 드러나야 한다.
가능 여부나 조건을 묻는 질문에는 가능/불가능과 조건이 함께 나와야 한다. `규칙이 있다`, `조건이 있다`에서 멈추면 이슈다.
`여기가 어디인지`, `인간은 무엇인지`, `출항 규칙은 무엇인지` 같은 질문에 답이 나와야 한다. 답변이 장식 문장으로 흐르거나 행동 가능한 정보가 빠지면 이슈다.
현재 위치 질문의 답은 정확한 장소명에서 시작해야 한다. `안개 항구`에 있는데 `안개 바다`, `붉은섬`과 같은 더 넓은 이름만 답하면 이슈다.
장소 이동 로그에는 `location.description`이 그대로 보인다. 이 문구가 `~다`, 명사형, 메모체로 되어 있어 주변 나레이션의 `합니다체`와 부딪히면 `ExposedDescriptionTone` 이슈다. 단, `mood`, `traits`, `knowledge.summary` 같은 내부 재료까지 억지로 존댓말로 바꾸지는 않는다.
선착장 같은 준비 장소는 현재 장소다. 그곳에 들어갔을 뿐인데 배가 이미 떠났거나 다음 섬 광장이 보이는 것처럼 쓰면 `NoPrematureState`와 `StagingPlace` 이슈다.
이동 직후 로그는 현재 장소의 인물과 사물만 행동 주체로 삼아야 한다. 직전 대화 상대가 새 장소에 없는데도 손짓하거나 노를 잡는 식으로 계속 등장하면 `CurrentActor` 이슈다.
이동 로그는 직전 대화의 질문/답변을 계속 요약하는 자리가 아니다. 도착 장소의 보이는 상태와 다음 선택으로 바로 넘어가야 한다.
quest handoff도 같은 기준을 적용한다. 이전 퀘스트 물건이 handoff에 없는데 다시 등장하거나, 선택 보상을 받지 않은 루트에서 물건이 플레이어 손에 올라오면 `HandoffLeak` 이슈다.
seed와 guidance로 막아도 직전 대화가 계속 섞이면 런타임 narration brief가 이동 장면에 `previous_scene`/`recent_exchanges`를 넘기는지 확인한다.
메인 루트 선택 직후의 다음 섬 이동은 실제 입력으로 확인한다. 연결이 여러 개인 장소에서 `주황 광장으로 이동한다`, `흰섬 앞 바다로 이동한다`처럼 목적지를 정확히 말했는데 첫 번째 일반 연결로 빠지면 `ConnectionPriority` 이슈다. 이런 경우 다음 루트로 나가는 gated connection을 로컬 되돌아가기 connection보다 앞에 둔다.
특정 NPC에게 묻는 턴은 대상자가 직접 답하거나 답을 피해야 한다. 다른 NPC가 대신 말하거나 플레이어의 질문문으로 끝나면 `TargetResponder` 이슈다.

필수 진행 조건은 판정 실패만으로 막히면 안 된다. 판정은 추가 정보, 신뢰, 더 좋은 힌트를 여는 데 쓰고, 튜토리얼의 필수 출구 자체를 잠그는 데 쓰지 않는 편이 좋다.
필수 메인 루트도 같은 기준을 적용한다. 핵심 선택지가 `social_check` 성공에만 열리면 운이 나쁜 플레이어가 같은 말을 반복하다가 막힌다. 주사위 판정은 정보량, 관계 변화, 더 좋은 단서를 주는 장면에 쓰고, 루트 진행 선택지는 핵심 장소 진입이나 명확한 관찰처럼 실패하지 않는 행동으로 열리는지 본다.

첫 갈등 장은 튜토리얼보다 더 깊게 눌러본다. 핵심 인물 각각에게 묻고, 중심 물건을 살피고, 군중이나 주변 반응을 확인하고, 반대 선택도 시도한다. 재미는 철학 설명보다 충돌 구조에서 나온다. 누가 요구하는지, 누가 빈손인지, 어떤 물건이 누구 손에 있는지, 기록이나 책임이 어떻게 갈리는지가 화면과 로그에서 계속 유지되어야 한다.

후속 섬도 첫 갈등 장의 밀도를 기준으로 확인한다. 새 섬에 도착하면 `왜 이런 규칙이 생겼는지`, `지금 누가 무엇 때문에 곤란한지`, `흰/검은 입장은 무엇을 지키려는지`, `장소에서 무엇을 볼 수 있는지`를 물어본다. 답이 철학 요약이나 최종 선택 설명으로만 나오고 생활 단서가 부족하면 `LaterChapterCoverage` 이슈다.

검은 보상 아이템은 선택 결과다. 선택 전 장소 조사나 이동 로그에서 이미 바닥에 놓인 보상처럼 보이거나, 플레이어가 주울 수 있는 물건처럼 노출되면 `RewardVisibility` 이슈다. 원인은 보통 `location.items`에 선택 보상 아이템을 미리 넣은 seed 문제다.

catalog, UI 라벨, fixed support record는 선택지와 이름을 보여주는 데이터다. `금지`, `목표`, 말투 지시, 장면 운영 규칙처럼 프롬프트에 가까운 문구가 catalog에 들어가야만 품질이 맞춰진다면 `DataPromptBoundary` 이슈다. 시나리오 고유 규칙은 seed guidance나 scenario 파일에, 공통 문체·런타임 규칙은 server prompt나 QA 기준에 있어야 한다.

나레이션 로그 품질을 볼 때는 다음을 함께 기록한다.

- 길이: 한 턴 로그가 화면에서 부담스럽지 않은가?
- 정보량: 플레이어가 다음 행동을 고를 만큼 이해했는가?
- 장면성: 보고서 문장이 아니라 짧은 동작, 거리, 자세, 반응으로 읽히는가?
- 상태 가독성: 성공/실패, 변화한 대상, 현재 위치·자세·위험이 장면 안에서 분명한가?
- 톤: 존댓말/합니다체, 시나리오 톤, NPC 말투가 맞는가?
- 구체성: `선택`, `운명`, `바다가 받아들이지 않음` 같은 추상어가 핵심 정보를 대체하지 않는가?
- 상태 정확성: 아직 배에 타지 않았는데 배 위에 있다고 쓰는 식의 시간 앞지름이 없는가?
- UI cue: 본문과 장면 변화/추천 버튼이 같은 사실을 가리키는가?

## 플레이 절차

1. 화면을 관찰하고 현재 상태를 짧게 기록한다.
2. 한 턴마다 실제 플레이어처럼 하나의 한국어 행동을 입력한다.
3. 확인 모달이나 판정 UI가 나오면 화면에 보이는 흐름대로 처리한다.
4. 매 턴 후 다음을 기록한다.
   - 입력
   - 관찰된 GM 반응 또는 UI 반응
   - 바뀐 상태
   - 헷갈린 점 또는 막힌 점
   - 콘솔 에러나 시각적 문제
5. 같은 문제가 반복되면 더 파고들기보다 재현 절차를 안정화한다.
6. 치명적인 진행 불능, 저장 손상, 화면 전체 오류가 나오면 중단하고 그 지점을 리포트한다.

## 필수 회귀 관찰

최근 UI/런타임 변경 이후 아래 항목은 QA 중 보이면 함께 확인한다.

- 상단 칩: 긴 장소/NPC/상태 칩이 화면 오른쪽에서 잘리지 않고 줄바꿈 또는 축약된다.
- 상단 패널: `주인공`, `챕터` 패널이 상단에서 열리고 기존 로그를 밀어내지 않고 덮는다.
- 챕터 패널: 같은 요약이 중복 표시되지 않고, 퀘스트 포기 같은 중복 액션이 없다.
- 할일/퀘스트: 완료된 퀘스트는 목록에서 숨겨지고, active/pending 퀘스트는 상태에 맞는 버튼만 보인다.
- 로그: 주사위 결과나 시스템 로그에 불필요한 왼쪽 막대가 없고, 본문 크기가 유저 채팅과 어울린다.
- 레벨업: HP/MP/스탯뿐 아니라 기술 습득/강화 선택지가 나오고, 오류 문구(`Failed to fetch`)가 보이지 않는다.
- 엔딩: 필수 퀘스트 완료 후 `이야기 완료` 상태가 보이고, 오래된 주변 행동 버튼이 남지 않는다.

## 플레이 스타일

- 일반 플레이어가 할 법한 행동을 한다.
- 한 턴에 여러 의도를 섞지 않는다.
- UI에 보이지 않는 내부 정보를 사용하지 않는다.
- 숨은 정답을 일부러 맞히려 하지 않는다.
- 한국어 2인칭 존댓말, 기술 용어, 퀘스트/전투/판정 흐름을 유심히 본다.
- 화면상 선택지가 있더라도 자유 입력이 가능한지 주기적으로 확인한다.
- 초반부 QA에서는 숨은 정답을 아는 사람처럼 진행하지 않는다. 먼저 현재 화면에서 보이는 단서로 질문하고, 그 답이 다음 행동으로 이어지는지 본다.

## 이슈 분류

각 이슈는 아래 중 하나 이상으로 분류한다.

- `server`: 런타임 규칙, 나레이션, 판정, 저장, API 응답 문제
- `client`: UI 표시, 입력, 모달, 상태 반영, 콘솔 에러 문제
- `scenario`: seed 내용, 단서, NPC, 퀘스트, 공개/비공개 정보 문제
- `ux`: 플레이어 이해, 다음 행동 추론, 피드백, 복귀성 문제
- `unknown`: 표면만으로 원인을 좁히기 어려운 문제

심각도:

- `P0`: 진행 불능, 데이터 손상, 화면 전체 오류
- `P1`: 핵심 플레이 루프가 깨지거나 잘못된 결정을 유도
- `P2`: 반복 혼란, 중요한 정보 누락, 불편한 우회 필요
- `P3`: 문구, 작은 시각 문제, 낮은 빈도 개선점

## 리포트 형식

대화 중 보고는 파일 리포트처럼 쓰지 않는다. 사용자가 바로 판단할 수 있게 짧고 자연스럽게 말한다.

- 괜찮은 점과 걸리는 점을 먼저 말한다.
- `정상/비정상`보다 `괜찮다`, `걸린다`, `아직 확정하기 어렵다`처럼 플레이 감상에 가까운 말을 쓴다.
- 원인은 시드, 런타임, UI 중 어디가 의심되는지만 짧게 붙인다.
- 같은 검증 문구를 반복하지 않는다.

## 금지

- 브라우저 QA 중 발견한 문제를 사용자 요청 없이 바로 수정하지 않는다.
- 사용자가 "고칠 건 바로 고쳐"라고 명시한 경우에는, 먼저 브라우저 증거와 재현 절차를 확보한 뒤 작은 수정만 적용하고 같은 브라우저 흐름으로 재검증한다. 원인 분석 단계에서는 코드·테스트·API 확인을 사용할 수 있지만, 최종 이슈 근거는 화면 관찰이어야 한다.
- 내부 API 호출만으로 진행하고 UI 관찰을 생략하지 않는다.
- 화면에 없는 숨은 상태를 확정 사실처럼 쓰지 않는다.
- 턴 번호 없는 이슈를 만들지 않는다.
- 브라우저 관찰이 아닌 추측을 이슈 근거로 쓰지 않는다.
