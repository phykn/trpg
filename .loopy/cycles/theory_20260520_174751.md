# UI Theory Cycle

## Target

Strengthen `theories/THEORY_UI.md` by deciding which cached-node affordances must be disabled when the backing node is not current in the latest server snapshot.

## Cycle number

2

## Prior survivor carried into the next cycle

The UI should use authority-preserving projection with bounded client continuity. Client continuity caches can preserve orientation, but stale cached facts must be demoted or disabled before becoming actions.

## Claim

Cached continuity nodes may remain visible for orientation, but any executable affordance that sends a structured `GraphAction` must be disabled unless the latest server snapshot marks that node reachable/current.

Cached nodes may still offer free-text suggestions only when phrased as exploratory proposals, not confirmed actions.

## Evidence

- `client/logic/game/useGame.ts` stores `storyGraph` after `mergeAndStoreStoryGraph`, so UI state may include cached continuity.
- `client/logic/story-graph/presenters.ts` demotes stale `place` and `subject` nodes, drops stale `item` nodes, and preserves other cached nodes.
- `client/logic/story-graph/_nodeActions.ts` creates structured graph actions from reachable movement/meeting/item node statuses.
- `client/components/story-graph/NeighborhoodPanel.tsx` derives selected actions from story graph nodes.

## Objection

The claim is too narrow because `GraphAction` is not the only executable affordance.

`PanelAction` with `kind: "text"` also initiates a server request through `onSend`. Cached quest nodes can become one-tap text actions such as inspect, and that still executes against the server. Therefore stale cached continuity can create executable authority even without sending a structured `GraphAction`.

## Revision

Cached continuity nodes may remain visible for orientation, but no one-tap affordance that initiates a server request may be derived solely from cached continuity. This includes structured `GraphAction`, quest/combat actions, and generated free-text `PanelAction`s.

Executable affordances must be built from the latest server-current decision surface, or from cached nodes that have been explicitly revalidated as reachable/current in that surface.

Cached-only nodes may show non-executable orientation text. If they offer language for the composer, it must be clearly exploratory and require ordinary user submission rather than a preconfirmed one-tap action.

## Revision target

theory

## Why it is stronger

It identifies execution by effect, not by payload type. The boundary now covers both structured actions and one-tap free-text requests, which matches how the client actually sends actions.

## Decision

Promote into `theories/THEORY_UI.md`.

## Named weak points and decisions

- Only `GraphAction` can create stale executable authority: rejected.
- Cached quest/text actions are harmless because classify revalidates server-side: narrowed. Server revalidation helps, but the UI must not present cached-only facts as one-tap executable affordances.
- Cached-only composer language: allowed only as non-executable draft/exploration, not direct submit.

## Progress toward target

The UI theory now has a stronger executable-affordance boundary.
