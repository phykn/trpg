# SERVER Theory Cycle

## Target

Define the minimum architecture rule for optimizing the server codebase without weakening TRPG playability, persistence, or LLM boundaries.

## Cycle number

1

## Claim

The optimal server architecture is graph-authoritative transition orchestration: domain graph/rules own facts, engines produce `GraphChange` transitions, runtime orchestrates turn flow and LLM calls, wire/db adapt at boundaries, and API stays glue.

A server change is well-placed only if it preserves graph as relational SSOT, keeps LLM output as proposal/evidence rather than state, and verifies every player-visible transition against graph changes before narration/front payload.

## Evidence

- `server/AGENTS.md` defines the layer rule as `api -> game.runtime -> llm.calls/game.engines -> llm/wire -> db -> game.domain/game.rules`.
- `server/AGENTS.md` names the runtime `Graph` as the source of truth for relations, and requires relation reads through graph query helpers.
- `server/src/game/runtime/flow/turn.py` prepares an action, dispatches it, dirty-saves graph changes, commits logs/progress, builds narration, then returns `graph_to_front_state`.
- `server/src/game/runtime/action/dispatch.py` routes player actions to graph engines and applies graph changes.
- `server/src/wire/graph/to_front.py` projects `GameRuntimeState` into the front payload.

## Objection

The claim overstates `GraphChange` as the verifier for every player-visible transition.

The front payload is not only graph facts. It includes hero/place/quest/combat fields plus pending confirmation, pending roll, log entries, and scenario completion. Pending confirmation and roll state are progress records, not graph changes. Narration creates player-visible log/history/dialogue artifacts outside `GraphChange`.

The critic also found graph fact changes in roll flow that are dirty-tracked after direct runtime graph copying/mutation rather than produced by an engine-returned `GraphChange`. That makes "engines produce every GraphChange transition" too strict for the current codebase.

## Revision

The optimal server architecture is runtime-state-authoritative transition orchestration with graph-authoritative relational facts.

Domain graph/rules own durable world facts. Graph fact mutations should be planned as `GraphChange` transitions by engines, or by an explicitly bounded runtime transition that is equivalent to a graph-change plan and dirty-tracked. Runtime owns turn orchestration, LLM calls, and non-graph runtime state such as progress, pending confirmations/rolls, logs, history, and dialogue. Wire/db adapt at boundaries, and API stays glue.

A server change is well-placed only if graph-derived facts remain relational-graph SSOT, LLM output remains proposal/evidence/narration rather than authoritative world state, and every player-visible payload is derived from the committed runtime snapshot.

## Revision target

theory

## Why it is stronger

It keeps the graph SSOT boundary for relational facts while admitting that the server has other authoritative runtime records. It also turns current roll-flow exceptions into a decision rule: runtime-local graph mutations are acceptable only when explicitly bounded, dirty-tracked, and equivalent to planned graph transitions.

## Decision

Promote to `theories/THEORY_SERVER.md`.

## Named weak points and decisions

- "Every player-visible transition must be verified by GraphChange": rejected. Progress/log/narration state is authoritative outside graph changes.
- "Engines must produce all graph mutations": narrowed. Engines are preferred, but bounded runtime transitions can exist when dirty-tracked and equivalent to graph-change planning.
- "LLM output can update state": rejected. LLM output may propose actions, supply narration, or produce persisted narration metadata only through runtime-owned commit paths.

## Progress toward target

The target has a stable boundary: optimize server code by preserving committed runtime snapshot authority, graph SSOT for relations, and thin adapters.
