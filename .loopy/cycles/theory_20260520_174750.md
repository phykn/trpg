# SERVER Theory Cycle

## Target

Strengthen `theories/THEORY_SERVER.md` by deciding when bounded runtime graph transitions must move into `game.engines`.

## Cycle number

2

## Prior survivor carried into the next cycle

The server should use runtime-state-authoritative transition orchestration with graph-authoritative relational facts. Bounded runtime graph transitions are allowed only when explicitly bounded, dirty-tracked, and equivalent to graph-change planning.

## Claim

A bounded runtime graph transition can stay in runtime only when it is local to one orchestration-only flow, does not encode reusable game rules, is fully dirty-tracked, and cannot be triggered from more than one player action path.

Move it into `game.engines` when it encodes reusable rules, changes durable world facts shared by multiple flows, or needs focused engine tests independent of LLM/runtime orchestration.

## Evidence

- `server/AGENTS.md` says `game.engines` owns pure game logic, graph changes, growth, inventory, skill, quest, recovery, and invariants.
- `server/AGENTS.md` says `game.runtime` owns graph-native session orchestration.
- `server/src/game/runtime/flow/turn.py` is reused as the runtime commit wrapper around action dispatch, dirty save, logs, narration, and front-state projection.
- `server/src/game/runtime/flow/roll.py` currently chooses roll stat/DC, computes affinity-influenced DC, applies affinity and XP graph effects, and saves dirty graph ids.

## Objection

The "cannot be triggered from more than one player action path" condition is too strong.

Runtime orchestration is intentionally reused across direct actions, confirmations, and successful rolls. Reuse by multiple entry paths does not mean the transition belongs in engines when the reused part only commits state, logs, narration, persistence, and dirty tracking around game semantics planned elsewhere.

The real boundary is whether runtime is deciding reusable game-rule semantics. Roll flow currently does decide several such semantics: stat/DC choice, affinity-modified DC, relation effects, one-time XP awards, and quest/growth effects. Those are game behavior, not orchestration.

## Revision

A bounded runtime graph transition can stay in runtime when it coordinates session flow, pending state, LLM/narration, logs, progress commits, persistence, and dirty tracking around state changes whose game semantics are already planned elsewhere. It may be reused by multiple player entry paths if that reuse is orchestration-only.

Move the transition, or at least its planning function, into `game.engines` when it decides reusable game semantics: DC/stat selection, success/failure consequences, relation/node fact changes, quest/growth/reward effects, combat/inventory/recovery invariants, or any durable world fact that should be testable without runtime, repo, LLM, log, or wire concerns.

## Revision target

theory

## Why it is stronger

It rejects caller count as the boundary and replaces it with semantic ownership. This better matches the current architecture, where runtime commit wrappers are reused legitimately while roll behavior still exposes engine-worthy game rules.

## Decision

Promote into `theories/THEORY_SERVER.md`.

## Named weak points and decisions

- Multi-path triggering as engine boundary: rejected. It is only a smell when the shared code owns game semantics.
- Runtime graph mutation as always invalid: narrowed. Runtime can coordinate bounded commits, but reusable semantic planning belongs in engines.
- Roll helper semantics in runtime: carried forward as implementation pressure, not a theory exception.

## Progress toward target

The server theory now has a sharper runtime-vs-engine decision boundary.
