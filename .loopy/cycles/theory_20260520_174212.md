# UI Theory Cycle

## Target

Define the minimum architecture rule for optimizing the client UI codebase without duplicating server authority or losing playable continuity.

## Cycle number

1

## Claim

The optimal UI architecture is decision-state projection: services/wire owns transport, `useGame` owns authoritative game state and actions, logic domains derive decision-relevant panels, components render stable domain surfaces, UI primitives/tokens own visual consistency, and locale owns client labels.

A UI change is well-placed only if it renders server-composed game text verbatim, exposes only decision-relevant state/affordances, and avoids duplicating server authority in client state.

## Evidence

- `client/AGENTS.md` defines `services/` as the data boundary and `services/wire.ts` as the client network contract.
- `client/logic/game/useGame.ts` owns graph game state, actions, pending confirmation/roll handling, request flow, and state application from server responses.
- `client/logic/game/requestRunner.ts` handles request generations, aborts, optimistic/streaming logs, and final state application.
- `client/AGENTS.md` requires server-composed strings to render verbatim and client-owned labels to live in `locale/`.
- `client/AGENTS.md` assigns domain calculation to `logic/<name>/`, domain views to `components/<name>/`, and visual primitives/tokens to `components/ui/` and `design/tokens.js`.

## Objection

The claim is too strict about pure projection from the current authoritative server state.

The current client intentionally owns non-authoritative continuity state. `useGame` merges server `storyGraph` snapshots through `mergeAndStoreStoryGraph` before storing state. The client guide says newer server snapshots merge over cached graph data so transient absences do not blank the map. The story-graph presenter preserves older nodes but demotes stale places/subjects and drops stale items.

This matters because story graph nodes can produce actions. `NeighborhoodPanel` derives selected actions from nodes, and `_nodeActions.ts` turns reachable node states into graph/text actions. The safety condition is not "no client continuity cache"; it is "client memory cannot invent executable authority."

## Revision

The optimal UI architecture is authority-preserving projection with bounded client continuity.

`services/api` and `services/wire` own transport and contract adaptation. `useGame` owns authoritative game state application and server actions. Logic domains derive decision surfaces. Story graph may own explicitly non-authoritative continuity caches, but stale cached facts must be demoted or disabled before they become affordances. Components render stable domain surfaces. UI primitives/tokens own visual consistency. Locale owns client labels.

A UI change is well-placed only if it renders server-composed game text verbatim, exposes decision-relevant state or affordances, keeps server authority out of client-owned state, and marks client memory as continuity/display state that cannot invent executable authority.

## Revision target

theory

## Why it is stronger

It preserves the server authority boundary while explaining why the story graph cache exists. It forces a concrete UI decision: cached continuity can help orientation, but it must not create executable actions unless the latest server-derived state still authorizes them.

## Decision

Promote to `theories/THEORY_UI.md`.

## Named weak points and decisions

- "UI should be pure projection only": rejected. Story graph needs bounded client continuity.
- "Client memory is harmless display state": narrowed. It is harmless only when stale affordances are demoted or disabled.
- "Client can infer server authority from cached nodes": rejected. Server actions must remain rooted in current authoritative state or safe free-text proposal paths.

## Progress toward target

The target has a stable boundary: optimize UI by preserving server authority, allowing bounded continuity memory, and keeping visual/domain/transport ownership separate.
