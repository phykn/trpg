## 1. 배경 — 무슨 통증을 푸는가

### 사용자가 직접 말한 통증
> "서버랑 클라가 의존적이라 서버를 바꾸거나 (혹은 클라를 바꾸면) 게임 진행이 안 되는 게 가장 큰 통증."

> "메시지별로 메시지가 나타나는 순서가 있는데 이 코드들이 다 떨어져 있어서 어떤 메시지가 먼저 나오는지 파악하기 어렵고, 향후 다국어를 지원할 건데 이건 어떻게 할지도 의문."

### 통증을 두 갈래로 분리

| 통증 | 정체 | 푸는 도구 | 끝나는 시점 |
|---|---|---|---|
| **A. 양쪽 결합** | 서버는 dict 직렬화, 클라는 손작성 TS — 자동 동기화 없음. 한 필드 바꾸면 양쪽 전부 손봄 | 통역기 (구조 SSOT) | 모든 페이로드/SSE 이벤트가 모델로 정의 |
| **B. 메시지 분산 + 다국어 미정** | 한국어 274줄이 31파일에 분산. 어떤 메시지가 어디서 만들어지는지 추적 어려움 | locale (텍스트 SSOT) | 모든 한국어 문자열이 카탈로그로 빠짐 |

### 컨텍스트 (새 세션이 잃으면 곤란한 것)

- **AI가 코드를 보지만 인간 가독성도 통증.** 사용자가 직접 코드를 매일 보진 않더라도 위계가 흐트러지면 결정을 내릴 때 통증으로 돌아옴. 위계 정리는 6분할의 부산물 + 그 자체로 가치 (라운드 1·2 후 사용자 정정).
- **MECE 위계는 사용자가 명시적으로 원함** — AI가 잘 읽으려면 사람도 잡힐 위계가 필요하다는 결.
- **"당장 뭘 할 건 아니다"** — 이 문서는 plan 자료. 실행 시작 시점은 사용자 결정.

## 2. 큰 그림: 서버 6분할

| 분할 | 역할 | 단일 진실 | 흡수 폴더 |
|---|---|---|---|
| **1. locale** | 텍스트 — 다국어 카탈로그, 시스템 메시지 빌더, 도메인 라벨, LLM 프롬프트의 한국어 자료 | 텍스트 SSOT | `locale/` |
| **2. llm** | LLM 호출/runner/schema + 프롬프트 + 컨텍스트 빌더. 프롬프트는 locale 카탈로그를 참조해 조립 | — | `llm/`, `llm_calls/`, `context/` |
| **3. game** | 도메인 + 규칙 + 엔진 + 관계 SSOT + 플로우 | 관계 SSOT (`ontology/graph.py`) | `domain/`, `rules/`, `engines/`, `ontology/`, `flow/` |
| **4. wire (통역기)** | 서버↔클라 통역 — Pydantic 모델, emit 빌더, codegen, mapping orchestrator | 구조 SSOT | `wire/`, `mapping/` |
| **5. api** | FastAPI 라우터 + SSE 엔드포인트 + 인증/CORS 글루 | — | `api/` |
| **6. persistence** | Supabase + LocalFs adapter, SaveRepo/ScenarioRepo Protocol | — | `persistence/` |

### 6분할 결정 근거 (다시 안 돌게)

- 사용자가 그린 3분할(LLM / 게임 시스템 / 송수신)에 통역기를 4번으로 분리. 송수신과 통역기는 한 자리로 시작.
- 통역기는 **서버 안에 둠 (옵션 A)** — 사용자 표현: "통역기 부분도 그냥 서버가 마스터로 해서". 형태 잡힌 후 별도 패키지(옵션 B)로 옮길 여지는 남김. 옮길 신호는 "통역기 안 모델을 서버 로직 때문이 아니라 클라 표시 때문에 고치는 비율이 의미 있게 올라올 때".
- locale을 별도 분할로 둠 — 사용자 표현: "이 프로젝트가 언어를 다루는 거니까 언어에 대한 부분은 한 부분에서 관리되면 좋겠다."
- `flow/`는 game 안에 둠 (turn orchestration이 게임 로직). 통역기·locale의 일이 빠지면서 자연스럽게 얇아짐.
- **4 → 6분할 갱신 (2026-05-05, 라운드 2 finale 후):**
  - **api/와 wire/ 분리:** wire는 payload shape + emit 빌더 + codegen (구조 SSOT). api는 HTTP/SSE 표면. 의존 방향이 `api → wire` 단방향이고 책임 결이 다름. 같은 폴더로 묶으면 "wire" 이름이 두 의미를 동시에 가져 명명 혼란.
  - **persistence/ 1급 분리:** `SaveRepo`/`ScenarioRepo` Protocol seam이 Supabase·LocalFs 어댑터를 결합한 단단한 분리체. layer rule `api → flow → ... → persistence → domain/rules`상 명시적인 1급 자리. game/ 안 묻으면 layer 의미 흐려짐.
  - 라운드 1·2 결과 wire/는 자기 자리 잡고 mapping/이 슬림해진 지금, api/persistence 분리는 추가 churn 거의 없이(폴더 이름·이동만) 더 정직한 layer를 표현.

### CLAUDE.md 기존 원칙과의 정합성

- "Display data is built on the server and shipped over" — 유지. locale 카탈로그가 서버에 살고 서버가 빌드.
- "Korean only" — 유지. 단 카탈로그 도입으로 향후 locale 분기가 가능해짐.
- 레이어 룰 `api → flow → llm_calls/engines → llm/ontology/context/mapping → persistence → domain/rules` — 큰 충돌 없음. 통역기는 `mapping/` + `api/` 자리에서 자라남, locale은 새 layer.

## 3. 다국어 모델 결정: W (풀 i18n + 한 곳 관리)

검토했던 옵션:
- ~~X. 서버 빌드(현재 구조 유지)~~ — 분산 통증 안 풀림.
- ~~Y. 클라 빌드~~ — CLAUDE.md "서버가 빌드" 원칙 뒤집힘. 거부.
- ~~Z. 하이브리드 (LLM은 한국어 고정)~~ — 사용자가 거부. 표현: "LLM 프롬프트 자체는 정해진 언어로 출력하라고 하면 될 거 같고 (클라에서 설정된 값으로) 안에 지원되는 언어에 대한 정보를 넣으면 될 것 같음."
- **W. 풀 i18n** — 채택.

### W의 형태

- LLM 프롬프트도 locale 카탈로그에서 조립. `_kernel.md` 같은 자리에 "Output language: {locale}" 지시 + 도메인 어휘를 카탈로그 키로 끌어옴.
- 시스템 메시지/도메인 라벨/조사 처리도 카탈로그 단일 진실.
- 카탈로그 형태: `key → {ko, en, ja, ...}` + 변수 보간 + 한국어 조사 처리 (현 `mapping/josa.py` 결을 어댑터로).
- 클라 i18n 모드(서버 대신 클라가 직접 카탈로그 받아 빌드)는 **지금 결정 안 함** — 필요해질 때 추가.

### W의 미묘한 점 (지금 결정 안 함, 미루기로 명시)

- **LLM 다국어 품질**: 모델별 비한국어 품질 차이 있음. 카탈로그·프롬프트 다국어화는 진행하되 "어느 모델로 어느 locale" 결정은 측정 후 별도 토픽.
- **시나리오 자료 다국어**: `scenarios/<profile>/world.md`, character description 등 한국어 자료. 시나리오 다국어 작성은 별개 큰 작업. 한국어로 두고 출력만 다국어로 변환하는 모드도 가능.

## 4. 첫 케이스: 에러 메시지 한 종류 통과

### 무엇을 하나
예: "인벤토리 가득 참" 류 작은 에러 한 종류를 새 분할 위에서 통과시킴. 이 한 번이 통증 A·B 양쪽 인프라(통역기 + locale)를 같이 깖.

### 왜 이 케이스인가 (다시 안 돌게)
- 후보 2(`pending_check`, 13키 페이로드)도 검토했으나, locale 측 검증이 약함 (라벨 한두 개만 닿음). 통증 A 인프라는 깔리지만 통증 B 인프라는 따로 깔아야 함.
- 후보 1(에러 한 종류)은 작은 페이로드 + 메시지 본문 둘 다 닿아 양쪽 인프라를 한 번에 깖.
- 다른 에러로 확장이 자연스럽고, 동시에 다른 페이로드 추가도 자연스러움.

### 작업 항목 (5단계)

1. **locale 카탈로그 자리 신설**
   - 위치 후보: `server/src/locale/`
   - 카탈로그 한 줄 등록: `error.inventory_full = {ko: "가방이 가득 찼습니다", en: "Inventory is full"}`
   - 변수 보간 함수 + 조사 처리 어댑터 (`mapping/josa.py`를 흡수하거나 호출)
2. **통역기 wire 모델 자리 신설**
   - 위치 후보: `server/src/wire/`
   - `ErrorPayload` Pydantic 모델 정의 — 필드 `code: str`, `message: str` (필요 시 `vars` 추가)
3. **빌더 함수 한 곳**
   - 시그니처 안: `make_error(code: str, locale: str, **vars) → ErrorPayload`
   - 카탈로그 조회 + 변수 보간 + 페이로드 채우기를 한 자리에서
4. **자동 생성 파이프라인 한 단계**
   - Pydantic `model_json_schema()` → JSON Schema → `client/types/`에 `.d.ts` 자동 생성
   - 클라 빌드 스크립트 한 줄 추가 (`npm run gen` 등)
   - 위치는 `server/src/wire/` 안에 schema 추출 스크립트 + `client/scripts/` 안에 변환기 한 줄
5. **현재 에러 발사 지점 한 곳을 빌더로 교체**
   - 예: `flow/error_phrases.py`의 인벤토리 가득 참 처리를 `make_error("inventory_full", locale, ...)` 호출로 교체
   - 클라는 자동 생성된 `ErrorPayload` 타입으로 받음 — 손작성 타입 하나 사라짐

### 한 번 통과로 검증되는 인프라 4종

| # | 검증되는 것 |
|---|---|
| 1 | locale 카탈로그 형태 (다국어 표 + 변수 보간 + 조사) |
| 2 | 통역기 wire 모델 형태 (Pydantic 정의 → 직렬화) |
| 3 | 빌더 함수 형태 (locale + 통역기 만나는 자리) |
| 4 | 클라 타입 자동 생성 파이프라인 |

### 이 케이스가 검증하지 못하는 것 (인지하고 시작)

- 큰 페이로드 (13키 이상)에서 자동 생성이 잘 작동하는가 — `pending_check` 통과 시 검증
- discriminated union (SSE 이벤트 타입) 자동 생성이 잘 작동하는가 — SSE 이벤트 통과 시 검증
- LLM 프롬프트의 카탈로그 참조 — 프롬프트 통과 시 검증

## 5. 첫 케이스 이후: 두 갈래 동시 진행

```
[에러 한 종류 통과] (인프라 4종 깔림)
   │
   ├─ 통증 A 갈래 (양쪽 결합 끊기) ─────────────────────
   │    pending_check → hero → subject → quest → place
   │    → SSE 이벤트들 (judge, narrative_delta, log_entry,
   │       combat_start/turn/end, suggestions, done)
   │    각각: wire 모델 한 개 + 빌더 한 줄
   │
   └─ 통증 B 갈래 (텍스트 단일 진실) ────────────────────
        다른 에러들 → flow/format.py 메시지들
        → flow/error_phrases.py → mapping/labels.py · josa.py
        → domain/types.py · clock.py 라벨
        → rules/config.py 라벨 → engines/* 한국어
        → LLM 프롬프트 (_kernel.md + 각 agent prompt)
        각각: 카탈로그 키 한 줄 + 호출 한 줄
```

### 두 갈래 진행의 부산물 — 위계 정리가 자동으로 따라옴

지금 `flow/`가 세 가지 일을 한꺼번에 함:
- (game system 일) turn orchestration
- (통역기 일) `yield {"type": ..., "data": ...}` SSE 발사
- (locale 일) 한국어 메시지 빌드

분할 진행이 끝나면:
- `flow/turn.py` (현 932줄), `flow/format.py` (현 401줄), `flow/error_phrases.py` (현 104줄) 등이 자연스럽게 얇아짐
- `flow/`는 orchestration만 남고 game 안에 자리잡음
- `mapping/to_front.py` (현 332줄)는 wire로, `mapping/labels.py` · `mapping/josa.py`는 locale로 갈라짐

→ 위계 MECE 정리는 **별도 작업이 아니라 6분할 진행의 부산물 + 그 자체로 인간 가독성 통증을 풂**.

|---|---|---|
| 통역기를 별도 패키지(옵션 B)로 옮기기 | 형태 잡히기 전엔 가치 약함 | 통역기 안 모델을 서버 로직 때문이 아니라 클라 표시 때문에 고치는 비율이 의미 있게 올라올 때 |
| 다국어 LLM 품질 측정·다른 모델 교체 | 카탈로그·프롬프트 다국어화 후 별도 토픽 | 첫 다국어 locale을 실제로 추가할 때 |
| 시나리오 자료 다국어화 | 큰 별도 작업 | 다국어 정식 출시 검토 시 |
| 폴더 재배치(MECE) 자체 | 두 갈래 진행의 부산물로 자연스럽게 따라옴 | 갈래 진행이 충분히 끝났는데도 위계가 안 잡혔다고 느낄 때 |
| 클라 측 i18n 모드 (서버 빌드 → 클라 빌드 전환) | 지금은 서버 빌드로 충분 | 클라가 동적으로 locale 전환해야 할 때 |
| `pending_check`, `hero` 등 큰 페이로드를 첫 케이스로 | 통증 B 인프라 검증이 약함 | 첫 케이스(에러) 통과 후 통증 A 갈래 첫 항목으로 |

---

## 8. 다음 단계 (실행 시작 시)

1. 첫 케이스로 정한 "에러 한 종류" 후보를 하나 정함 (예: `inventory_full`, 또는 `flow/error_phrases.py`에 있는 다른 작은 에러).
2. 위 §4 작업 항목 5단계대로 진행. 특히 단계 4(자동 생성 파이프라인)는 통과 후 다른 케이스에서 그대로 재사용되므로 형태를 깨끗하게.
3. 한 번 통과되면 인프라 4종 형태가 굳음. 그 다음부터는 §5의 두 갈래를 동시 진행 — 카탈로그 한 줄 + 모델 한 개 + 빌더 호출 한 줄 수준의 반복 작업.

## 9. 합의 굳음 — 새 세션에서 흔들지 말 것

다음은 이미 결정 끝남. 새 세션에서 다시 옵션을 던지지 말고 이 위에서 작업.

- 서버 6분할: locale / llm / game / wire(통역기) / api / persistence (4 → 6 갱신, 라운드 2 finale 후)
- 통역기 위치: 서버 안 (옵션 A). 별도 패키지(B)는 형태 잡힌 후.
- 다국어 모델: W (풀 i18n + 한 곳 관리)
- 첫 케이스: 에러 메시지 한 종류

다음은 결정 미정 — 새 세션에서 정함:

- 첫 케이스에 사용할 구체적 에러 종류
- locale·통역기 폴더의 정확한 이름 (`server/src/locale/`, `server/src/wire/` 권장이지만 사용자 취향)
- 자동 생성 파이프라인의 구체적 도구 (`openapi-typescript` 또는 직접 변환기)
