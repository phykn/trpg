# Theory Cycle UI 4

## Claim

`Situation`은 플레이어가 지금 어떤 상황인지 알 수 있게 해야 한다.

최소한 장소, 눈앞의 중요한 대상, 현재 위험이나 제약, 이전 선택 때문에 남아 있는 결과를 보여야 한다.

## Evidence

- 같은 방, 같은 문, 같은 경비병이라도 플레이어가 "몰래 지나간다", "협박한다", "함정을 찾는다" 중 무엇을 하려는지에 따라 중요한 상황 정보가 달라진다.
- 보이는 상태를 적게 보여주면 행동공간이 흐려진다. 하지만 많이 보여주면 숨겨진 정보가 누설되거나 플레이어가 이야기보다 상태표를 관리하게 된다.
- 관찰 가능한 정보라도 다음 행동 선택, 위험 판단, 결과 납득에 영향을 주지 않으면 핵심 상황이라기보다 부가 묘사에 가깝다.

## Objection

`Situation`을 visible state 목록으로 정의하면 TRPG UI의 중요한 기능을 놓친다.

플레이어에게 필요한 것은 세계 상태 전체의 열람이 아니라, 지금 선택을 바꿀 만큼 중요한 단서, 제약, 위험, 불확실성의 윤곽이다.

또한 "보이는 상태"가 곧 "의미 있는 상황"은 아니다.

의미 있는 상황은 플레이어의 다음 선택과 연결될 때 생긴다.

## Revision

`Situation`은 상태 목록이 아니라 결정 관련 프레이밍이다.

```text
Situation =
DecisionRelevantFrame
```

보여야 하는 정보는 다음 질문으로 판단한다.

```text
이 정보가 플레이어의 다음 행동 선택, 위험 판단, 결과 납득 중 하나를 바꾸는가?
```

바꾸지 않으면 핵심 상황 정보가 아니다.

바꾸지만 캐릭터가 알 수 없거나 추론해야 하는 정보라면 직접 공개하지 말고 징후, 불확실성, 질문 유도 형태로 보여야 한다.

## Why it is stronger

수정된 claim은 상황 정보를 양으로 판단하지 않는다.

대신 어떤 정보가 플레이 가능성에 기여하는지 묻는다.

이 기준은 범용 UI 설계와 코드 개선에 더 직접적으로 쓸 수 있다. 화면에 상태를 더 많이 보여줄지보다, 어떤 상태가 선택과 결과 이해에 필요한지 먼저 판단하게 하기 때문이다.

## Next question

세 항목이 모두 정교해지면서 원래 공식이 너무 표면적인 이름이 되었는지 확인해야 한다.

`Situation + ActionSpace + Feedback`을 유지해도 되는가, 아니면 `DecisionFrame + PlausibleAttempts + AccountableConsequence`처럼 더 정확하지만 무거운 이름으로 바꿔야 하는가?
