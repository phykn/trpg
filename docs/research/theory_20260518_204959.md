# Cycle 1: Minimum Conditions for Play

## Claim

An LLM-TRPG should preserve the minimum state needed to keep the game playable.

Formula:

```text
Minimize context; preserve playability.
```

## Evidence

- `README.md:3` says the LLM classifies player intent and writes narration, while the engine handles state, rules, rolls, and time. This supports a split between expressive text and play-preserving machinery.
- `server/AGENTS.md:43-47` defines the graph as runtime source of truth, prompt context builders as downstream views, and runtime as graph-native orchestration.
- `server/AGENTS.md:54-58` says relation questions must go through graph query helpers and writing must return or apply `GraphChange`, not mutate prompt/client payloads as state.
- `server/AGENTS.md:67-74` stores graph nodes/edges, progress, logs, history, and dialogue separately, implying not every remembered fact has the same status.
- `server/tests/game/runtime/test_llm_context_guards.py:185-208` asserts classify and narration payloads stay under 12000 characters even with many visible characters, inventory items, skills, and dialogue turns.
- `server/tests/game/runtime/test_memory_context.py:84-95` limits classify dialogue memory and prevents turn log from leaking into that payload.
- `server/tests/game/runtime/test_memory_context.py:119-140` selects same-target dialogue first, then fills with recent dialogue, showing continuity context is relevant but filtered.

## Objection

"Minimum state" is too narrow. The project does not only preserve graph state; it also preserves bounded progress, logs, turn summaries, and recent dialogue. Some of these are not graph facts, but they still keep narration and classification coherent.

If the theory says only graph state matters, it cannot explain why compact recent dialogue, related memory, pending rolls, pending confirmations, and logs exist.

## Revision

The theory should target minimum conditions, not minimum graph state.

Revised claim:

An LLM-TRPG should preserve the minimum conditions that keep play coherent: enough state and bounded context to preserve valid next actions, unresolved obligations, and continuity.

Formula:

```text
MinConditions = State + BoundedContext
```

## Why It Is Stronger

The revision still decides what to keep or reject:

- Keep data that changes valid next actions, unresolved obligations, or continuity.
- Bound data that helps continuity but does not belong to authoritative state.
- Reject or summarize data that does not affect playability.

It also explains why the graph is source of truth for relations while recent dialogue and related memory are still useful as bounded context.
