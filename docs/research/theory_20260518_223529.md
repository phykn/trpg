# Theory Cycle UI 5

## Claim

세 항목이 정교해졌으므로 `Situation + ActionSpace + Feedback` 대신 더 정확한 이름을 써야 할 수 있다.

후보:

```text
PlayableUI =
DecisionFrame + PlausibleAttempts + AccountableConsequence
```

## Evidence

- 이전 사이클의 반례는 이름이 틀렸다는 것보다 단순명을 얕게 해석하면 안 된다는 것을 보여준다.
- `ActionSpace`는 버튼 목록이 아니고, `Feedback`은 진짜 원인 공개가 아니고, `Situation`은 상태 목록이 아니다.
- UI 개선 기준으로는 단순 용어가 화면 검토 질문으로 바로 바뀐다. 무엇이 보이는가, 무엇을 시도할 수 있어 보이는가, 결과를 어떻게 알게 되는가.

## Objection

정확한 이름으로 바꾸면 정밀도는 올라가지만 공개 이론으로는 너무 무거워진다.

또한 하위 정의 일부만 대표명처럼 올라오면서 다른 조건이 흐려질 수 있다.

예를 들어 `AccountableConsequence`는 `ProcessingAcknowledgement`와 `StateDelta`를 덜 드러낸다.

`PlausibleAttempts`도 관찰된 상황에서 나온다는 제약을 이름만으로 충분히 말하지 못한다.

## Revision

공개 공식은 단순명을 유지하되, 괄호 정의를 붙인다.

```text
PlayableUI =
Situation(decision-relevant)
+ ActionSpace(plausible attempts)
+ Feedback(accountable consequence)
```

단순명은 UI 검토의 입구로 쓰고, 괄호 정의는 오해를 막는 판정 기준으로 쓴다.

## Why it is stronger

수정된 공식은 말하기 쉽고, 동시에 얕은 해석을 막는다.

`Situation`, `ActionSpace`, `Feedback`은 화면과 코드에서 찾기 쉬운 이름이다.

괄호 정의는 다음 오해를 막는다.

- `Situation`은 상태 목록이 아니다.
- `ActionSpace`는 선택지 버튼 목록이 아니다.
- `Feedback`은 숨은 원인 공개가 아니다.

## Next question

이 공식은 한 번의 행동 루프에는 강하다.

하지만 장기 TRPG 세션의 UI에는 누적 기억, 이전 선택의 잔여 효과, 세션 재진입, 미해결 목표처럼 시간에 걸친 플레이 가능성도 필요하다.

`PlayableUI`에 시간적 연속성을 별도 조건으로 넣어야 하는가, 아니면 `Situation` 안에 포함하면 충분한가?
