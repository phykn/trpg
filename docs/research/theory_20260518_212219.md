# Theory Cycle 9

## Claim

The final minimal condition set is:

```text
MinConditions = TransitionState + ContinuityContext
```

`TransitionState` consists of Actor, Locus, Possession, Capability, Meter, Attribute, EffectState, Relation, Knowledge, Commitment, PendingProcess, and Binding.

`ContinuityContext` consists of PresentationMemory and NarrativeMemory.

## Evidence

- `server/src/game/domain/graph/models.py` and `server/src/game/domain/graph/validation.py` define generic graph nodes and edge invariants such as `carries`, `equips`, `knows_skill`, `uses_effect`, `applies_status`, `uses_action`, and `has_knowledge`.
- `server/src/llm/context/classify_view.py` derives visible targets, exits, inventory, equipment, skills, merchants, corpses, affordances, and references from runtime state.
- `server/src/llm/calls/classify/grounding.py` validates grounded actions against the available view ids.
- `server/src/game/domain/progress.py` stores `runtime_content`.
- `server/src/game/domain/content.py` defines `RuntimeContent` records and lets `node_value` fall back from graph node properties to static content records.

## Objection

The top-level split survives, but the primitive list overclaims.

The evidence proves that the game needs authoritative transition facts, but it does not prove that every named bucket is an irreducible primitive. Some are current implementation shapes: ownership, capability, mutable values, effects, and status modifiers.

There is also a boundary leak: durable authority is not only graph and progress. Some authoritative facts live in content records referenced by graph nodes.

## Revision

Keep the top-level formula, but demote the long list from final primitives to current subconditions:

```text
MinConditions = TransitionState + ContinuityContext
```

`TransitionState` means authoritative graph, progress, and content facts needed for grounded action, valid transition, process control, and durable world truth.

Current `TransitionState` subconditions:

- Entity and locus facts
- Ownership and access facts
- Capability and action-affordance facts
- Mutable meters and attributes
- Rule-bearing effects, statuses, and modifiers
- Relations
- Knowledge and reveal facts
- Commitments
- PendingProcess
- Binding

`ContinuityContext` remains bounded narration-authoritative context:

- PresentationMemory
- NarrativeMemory

`ContinuityContext` may constrain narration, but cannot authorize grounded transitions by itself.

## Why it is stronger

The revision keeps the useful decision boundary while avoiding a false finality claim. It also accounts for `RuntimeContent` as an authority source when content-backed records decide actions, rules, truth, or narration constraints.

## Next question

Where exactly do `RuntimeContent` facts belong: inside `TransitionState`, outside `MinConditions` as static schema, or split by whether each fact affects transitions or narration?
