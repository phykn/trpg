# Cycle 2: Categories of Minimum Conditions

## Claim

`MinConditions` consists of eight categories:

- actor condition
- location condition
- possession condition
- relation condition
- knowledge condition
- obligation condition
- pending-resolution condition
- continuity context

## Evidence

- `server/src/wire/models.py:91-142` exposes hero, inventory, equipment, skills, place exits, place items, and visible targets to the client.
- `server/src/wire/models.py:167-217` exposes combat state, pending confirmation, pending roll, and log as front-state payload.
- `server/src/game/domain/progress.py:12-23` stores player id, active subject, active quest, turn count, pending confirmation, pending roll, combat state, and next log id.
- `server/src/llm/context/classify_view.py:20-32` defines explicit context limits for visible targets, exits, inventory, skills, location items, recent dialogue, merchant stock, corpses, and corpse items.
- `server/src/llm/context/classify_view.py:60-109` splits classify context into `identity`, `affordances`, `references`, and `budget`.
- `server/src/llm/context/classify_view.py:80-90` directly lists next-action affordances: speak, attack, move, use, pick up, and accept/abandon quest.
- `server/src/game/runtime/narration/memory_context.py:12-23` selects important history with a limit.
- `server/src/game/runtime/narration/memory_context.py:34-43` limits classify dialogue context.
- `server/src/game/runtime/narration/memory_context.py:88-114` selects related memory by related ids and importance with a limit.

## Objection

The eight categories mix different levels.

Actor, location, possession, relation, obligation, and pending-resolution are state-like. Continuity context is not the same kind of thing; it is a bounded view over logs, dialogue, and memory. Knowledge is ambiguous: some knowledge can be authoritative state, but current evidence mostly shows it as selected context or scenario content, not a clearly separate runtime condition.

The category list also misses a more central concept already present in code: affordance. Classify context explicitly computes possible next actions. That suggests the categories should be organized by whether they preserve affordances, not by whether they name familiar RPG nouns.

## Revision

`MinConditions` has two layers:

```text
MinConditions = AffordanceState + ContinuityContext
```

`AffordanceState` is authoritative data needed to determine valid next actions and unresolved resolutions.

Initial subconditions:

- actor: who can act, and whether they can act
- place: where actors/items are, and where movement can go
- possession: what can be used, equipped, transferred, or picked up
- relation: who can be targeted, spoken to, attacked, traded with, or otherwise related
- obligation: active quests, commitments, or goals that change available actions
- pending-resolution: confirmation, roll, combat, or other unresolved process that blocks or shapes the next input

`ContinuityContext` is bounded non-authoritative context used to keep classification and narration coherent.

Initial subconditions:

- recent dialogue
- related memory
- important history
- visible narration/log tail

Knowledge remains unresolved: it may become either affordance state or continuity context depending on whether it changes available actions.

## Why It Is Stronger

The revision compresses the category list into two concepts that decide storage and prompt boundaries:

- If it decides valid next actions or unresolved process, it belongs to `AffordanceState`.
- If it helps the LLM preserve continuity but should be bounded, it belongs to `ContinuityContext`.
- If knowledge changes affordances, it becomes state; otherwise it remains context.

This better matches the code split between graph/progress state and limited LLM context payloads.
