# LLM TTRPG 엔진 설계 (backend)

> 북극성 설계 노트. 시스템이 어떤 모양이고 왜 그렇게 잡았는지를 모아둔 곳. 할 일 목록이 아니라 들춰 보는 용도.

## 1. 설계 철학

### 무엇을 LLM 이 하고 무엇을 엔진이 하나

| 일 | 누가 |
|---|---|
| 이야기 풀어 쓰기 | LLM (내러티브 에이전트) |
| 행동 난이도 등급 매기기 | LLM (DC판정 에이전트) |
| 난이도 숫자 (DC) 계산 | 엔진 |
| 매 턴 LLM 에 줄 정보 묶음 만들기 | 엔진 (관계 그래프 따라 모음) |
| 게임 상태 갱신 | 엔진 |
| 전투 진행 | 엔진 |
| NPC 가 무엇을 할지 결정 | 엔진 (확률 규칙) |
| 호감도 변화량 계산 | 엔진 ([03-features.md](./03-features.md) §2.2) |
| 호감도가 주사위 결과에 주는 보정 | 엔진 ([02-runtime.md](./02-runtime.md) §5.4) |
| 성장·거래·장비·스킬·시간 처리 | 엔진 ([03-features.md](./03-features.md) §2) |

LLM 은 "무엇을 할까 / 어떻게 말할까" 만 고른다. 숫자, 상태, 규칙은 전부 엔진이 처리.

여기서 "에이전트" 는 한 번의 LLM 호출 단위. 매 턴 새로 부르고, 채팅 창처럼 이전 대화를 들고 있지는 않다 (왜 그렇게 했는지는 §3.5).

## 2. 단계 (Phase)

### 2.1 P1 — 기본 골격

- FastAPI (파이썬 웹 서버) 골격, 게임 세션 만들기·불러오기·저장 (JSON 파일에)
- 한 턴이 흐르는 순서: 난이도 판정 → (직접 굴려야 하는 분기면 잠시 멈춤 → 프론트가 주사위 굴림 → 결과 받음) → 이야기 쓰기 → 상태 반영
- 매 턴 LLM 에 줄 정보 묶음 (엔진이 미리 모아 줌):
  - `surroundings` (주변): 지금 어디에 있고, 거기 누가 같이 있고 무엇이 놓여 있는지. judge 가 액션 분류·대상 결정에 씀.
  - `target_view` (상대가 알 만한 것): 플레이어가 말 거는 대상이 자기 입장에서라면 알 만한 것 (성격, 임무, 플레이어를 어떻게 여기는지)
  - `history` (지난 흐름): 최근 턴 요약과 중요한 기억
- 게임 안 데이터 구조, 상태 전체를 담는 객체 (`GameState`), 안쪽 표현과 프론트로 나가는 표현 사이 변환
- SSE (Server-Sent Events — 서버가 프론트 쪽으로만 흘려 보내는 실시간 신호):
  - `judge`: 난이도 판정 결과 (행동 종류·등급)
  - `pending_check`: 잠시 멈춤 — 프론트에 주사위 버튼 띄우라는 신호
  - `narrative_delta`: 이야기 텍스트 조각 (한 글자씩 흐르는 중)
  - `suggestions`: 입력창 위에 띄울 행동 후보 칩 (0~3 개, 자유 입력 보조용)
  - `state`: 갱신된 게임 상태 한 묶음
  - `log_entry`: 로그 한 줄
  - `done`: 턴 끝
  - `error`: 처리 중 오류
- 새 게임 흐름:
  - 프로필 목록 (`GET /profiles`) → 사용자가 시나리오 카드 중 하나 고름. P1 시드는 `scenarios/default/` 한 벌만이지만, 디렉터리 스캔 방식이라 추후 시나리오를 폴더 추가만으로 확장 가능.
  - 캐릭터 생성 — 종족(목록 선택) + 외모(자유 텍스트 한 줄) + 이름. 스탯 6 개는 모두 10 으로 시작 (분배 화면 없음).
  - 골라진 race 의 racial_skills 자동 부여, max_HP/MP 는 [03-features.md](./03-features.md) §2.3 공식 (level 0).
- 마지막 게임 자동 복원 (`GET /session/current`): `SAVES_DIR/.current` 가 가리키는 game_id 의 `FrontState` 반환. 게임 목록·이어하기 화면 없음 — 한 명·한 게임 흐름.
- 기억·호감도·게임 안 시간 흐름은 최소한만 (등급·의도까지만 구현, 성향 보정은 P3 으로 미룸)
- 집/사무실 네트워크 안에서만 (환경 변수 누락 시 즉시 멈춤, 아이디·비밀번호 보호)

P1 안에 자리만 만들어 두는 약한 약속 — 작업 항목은 아니고, 구조만 노출해 놓고 LLM 이 글로 살릴지 알아서 판단:

- **날씨** (`Location.weather: list[str]`): "주변" 정보의 장소 부분에 노출. 엔진이 알아서 바꾸지 않고, 내러티브가 `set locations.{id}.weather` 로 직접 갱신.

### 2.2 P2 — 전투 (한 방 시네마틱)

- 전투는 한 d20 굴림으로 끝낸다. 라운드 루프가 아니라 한 방.
- 플레이어가 "고블린을 친다" 류로 입력 → judge 가 `combat` 액션으로 분류 → 엔진이 `pending_check kind="combat_roll"` 로 무장 → 프론트에서 주사위 버튼 → `/roll` 한 번으로 전투 전체 해결.
- 굴림 결과 등급 (critical_success ~ critical_failure) 이 기계적 결과 (사살 / HP 데미지 / XP / downed) 를 결정하고, `combat_narrate` 에이전트가 등급에 맞는 톤으로 5–10 문장 시네마틱을 스트리밍.
- 도주 (`flee`), 죽음 직전 판정 (death save) 도 같은 주사위 버튼 메커니즘 (`pending_check.kind="death_save"`) 으로 통일.
- SSE 신호:
  - `combat_start`: 전투 진입 알림 (turn_order, round, surprise, enemy_ids)
  - `combat_turn`: 시네마틱 안에서 한 actor 의 한 행동 (배경 정보용, UI 는 무시 가능)
  - `combat_end`: 전투 끝 — outcome 은 `victory / defeat / fled / downed / broken_off` 5 종
- 안쪽 상태 변경 종류 `death` (사망) 켜기

라운드 본문을 LLM 에 맡겨 한 라운드씩 굴리는 옛 안 (round-by-round) 은 룰을 지키느라 한 전투가 너무 늘어졌다. 한 방으로 합치고 결과 톤을 시네마틱이 살리는 쪽으로 바꿈 — 자세한 이유는 §3.15.

### 2.3 P3 — 확장

- 장비 입고 벗기, 가방 무게 제한, 거래 (사기·팔기·흥정)
- 성장 루틴 (레벨업 페어 트레이드, 스킬 학습)
- 스킬 시전, 지속 효과 (`ActiveBuff`) 매 턴 갱신
- 퀘스트 자동 진행 확인, 챕터·캠페인 전환
- 동반자 (companions) 시스템
- 위 모든 액션은 자연어로 한 통로 (`POST /turn`) 에 모이고, judge 가 종류를 분류 — 옛 메타 액션 REST 엔드포인트 (`/equip`, `/buy`, `/cast` 등) 는 폐기. 자연어 통합 결정의 이유는 §3.16.
- "거절된 행동" 기록을 보고 내러티브가 스스로 다음 글을 고치는 루프
- 게임 안 시간을 더 잘게 (이동·휴식·전투 매 턴별 흐름)
- 호감도 성향 보정 (질서 / 공격적 / 도덕)
- 외부 노출 (집/사무실 네트워크 밖에서도 접속 가능하게 풀기)

## 3. 왜 이렇게 정했나 — 결정 히스토리

### 3.1 왜 시그모이드 (S자 곡선) DC 인가?

기존 TTRPG (D&D 같은 종이·주사위 RPG) 는 직선식 DC 를 쓴다 — `20면 주사위 (d20) + 보정값 ≥ 난이도`. 직선식 장점은 플레이어가 머리로 계산하기 쉽다는 것. 이 게임은 엔진이 알아서 계산하므로 그 장점이 무뎌진다.

S자 곡선 (시그모이드) 의 장점:
- 능력치가 아무리 극단적이어도 결과가 자동으로 0~20 사이에 들어온다 (직선식은 별도로 잘라 줘야 함).
- 능력치가 난이도보다 훨씬 높아도 최소 1 은 굴려야 한다는 규칙이 자연스럽게 나온다 (기적·재앙 가능성을 남김).
- S자 모양은 양 끝이 평평하고 가운데가 가파른 곡선. 둘 다 약하거나 둘 다 강하면 작은 능력 차이는 잘 안 느껴지고, 비슷한 중간 구간에선 작은 차이도 결과가 크게 갈린다 — 사람이 능력 차이를 체감하는 방식과 닮았다.

### 3.2 왜 에이전트가 2개인가?

처음엔 3개로 잡았다 (난이도 판정 / 이야기 / 기록). 기록을 따로 빼면 관심사가 깔끔히 분리되긴 하는데:
- 이야기 에이전트는 이미 전체 맥락을 보고 있다. "이번 일을 어떻게 요약할지", "이건 따로 기억할 만한 일인지" 판단을 같이 시키는 게 질이 더 높다.
- LLM 호출이 1번 줄어든다 (반응 속도, 비용).

→ 기록을 이야기에 합쳐서 2개로.

### 3.3 왜 NPC 시점에서 `target_view` 를 만드나?

"LLM 에 뭘 줘야 할까" 의 기준이 필요했다. "이 경비병이 자기 입장에서 1인칭으로 안다면 뭘 알 만할까?" 로 잡으니 자연스럽게 범위가 정해진다:
- 경비병이 아는 것: 자기 성격, 임무, 플레이어를 어떻게 보는지, 최근 사건
- 경비병이 모르는 것: 플레이어 능력치, 다른 장소 상황, 플레이어 퀘스트 목록

→ 그 대상 (NPC, 장소, 물건) 이 알 만한 범위 = `target_view`. 게임 안 관계 그래프에서 1~2 칸 떨어진 노드만 따라가면 충분.

### 3.4 왜 한 턴에 한 판정만?

"설득하면서 동료가 금고를 턴다" 같은 두 판정 동시 처리는 막아 놓았다:
- 난이도 판정이 한 번에 한 결과만 반환하는 구조
- 두 판정을 자동으로 쪼개면 결과가 예측 불가
- 종이 TTRPG 도 한 턴에 한 판정이 표준

다만 **판정이 필요 없는 보조 동작은 한 턴에 같이 들어와도 된다** — "약초 먹고 검 든다" 같은 입력은 judge 의 `chain` 액션 (§3.16) 으로 묶여 sequential 로 처리. "한 턴에 한 판정" 은 주사위 굴림이 필요한 큰 행동이 한 번이라는 뜻이지, 모든 동작이 하나라는 뜻은 아니다. judge 는 절대 되묻지 않는다는 forward motion 원칙도 이 결정과 같은 결.

### 3.5 왜 매 턴마다 새 에이전트를 부르나?

Claude Code 같은 채팅 창에 의존하면 그 플랫폼에 묶인다. 매 턴 독립으로 부르면:
- 어떤 LLM 이든 갈아끼울 수 있다.
- LLM 의 컨텍스트 윈도우 (한 번에 받을 수 있는 글자 양) 한계와 무관 (필요한 것만 그때그때 쥐여 줌).
- 테스트·디버깅이 쉽다 (한 턴씩 따로 재현 가능).

대신 지난 흐름을 따로 챙겨 줘야 한다 → 히스토리 (지난 턴) + 메모리 (오래된 기억) 시스템.

→ [02-runtime.md](./02-runtime.md) §3.3, §7

### 3.6 왜 메모리에 "중요도" 를 다는가?

기억 개수에 상한 (예: N개) 을 걸어 놓고 단순히 오래된 순으로 지우면, "고블린 족장을 살려줬다" 같은 큰 사건이 사라질 수 있다. 중요도 기준이면 사소한 기억부터 정리된다.

### 3.7 왜 전투에선 엔진이 알아서 주사위를 굴리나?

매번 플레이어가 굴리면 전투가 번거로워진다. 평범한 공격은 엔진이 자동으로 처리하되, **무슨 행동을 할지는 매 턴 플레이어가 직접 고른다**.

특수 행동 ("샹들리에 떨어뜨리기" 같은) 은 난이도 판정이 `roll` (직접 굴리기) 로 분류해서 플레이어가 직접 굴린다.

→ [03-features.md](./03-features.md) §1.3

### 3.8 전투 NPC 의 행동 선택을 왜 LLM 이 아닌 확률로?

LLM 에 맡기면:
- 매 NPC 매 턴마다 LLM 호출 (비용·반응 속도 부담)
- 판단이 일관성 없게 흔들림

확률 규칙 엔진이면:
- 즉시 처리
- 어느 정도 예측되면서도 무작위성이 있다
- 설정 파일로 미세조정 가능

기본값은 가장 가까운 적 공격 70%, 무작위 30%. HP 낮으면 확률적으로 도망. 가중치는 전부 설정 파일에.

### 3.9 왜 CLI 에서 REST + SSE 로 바꿨나?

이전 버전은 `.venv/bin/python src/cli/main.py <명령>` 식 명령줄 도구였다. 빠른 동작 확인이나 디버깅엔 좋지만, 프론트 (React Native — 모바일 앱 프레임워크) 와 붙이려면:
- 네트워크 경계가 필요 (폰에서 백엔드 코드를 직접 실행할 수 없음)
- 글이 한 글자씩 흐르도록 보여 줘야 한다 (이야기 스트리밍)
- 상태 보기·바꾸기는 HTTP 의 표준 패턴 (GET/POST + JSON) 으로 처리되어야 자연스러움

→ FastAPI 라우트 (URL 진입점) 4개 + SSE. 옛 CLI 의 `scene` / `apply` / `roll` 은 각각 `GET /state` / 내부 `apply_changes` / `POST /roll` 로 흡수. 옛 CLI 는 더 이상 안 씀.

→ [02-runtime.md](./02-runtime.md) §2.4 (라우트 목록·SSE 신호)

### 3.10 왜 한 턴을 두 단계로 (`pending_check`)?

`roll` 분기는 "엔진이 난이도를 계산하면 → 플레이어가 주사위를 굴려야" 진행된다. 한 번의 HTTP 요청으로 처리하려면 길이 두 가지 있었다:

(a) WebSocket 양방향 — 프론트와 백엔드가 서로 신호를 자유롭게 주고받는 연결. 양쪽 다 "지금 어느 단계인지" 추적해야 (상태기계) 하고, 연결이 끊기면 복구가 까다롭다.
(b) 서버가 난이도 계산 후 무작위 주사위까지 알아서 굴려 결과를 한 번에 반환 — 플레이어가 "내가 굴린다" 는 감각이 사라진다.

대신 턴을 둘로 쪼갰다:
- `/turn` 이 `pending_check` (잠시 멈춤) 로 끝 → 프론트에서 주사위 버튼 UI 표시
- `/roll` 로 굴린 값 전달 → 이야기 이어서

상태는 서버의 `GameState.pending_check` 에 저장되므로, 연결이 끊겨도 `GET /state` 한 번이면 UI 가 복구된다. 프론트 `use-game` 훅 (React 의 상태 관리 단위) 은 정해진 단계 사이만 이동하도록 모델링 (`idle → streaming → awaiting_roll → streaming → idle`).

### 3.11 왜 게임 상태를 JSON 파일 한 덩이에 저장하나?

P1 은 사용자 한 명, 집/사무실 네트워크 안. 요구 규모가 작다:
- 한 게임 세션당 객체 수백 개 정도
- 초당 쓰기 빈도 낮음 (턴 단위)
- 동시 처리, 다중 사용자, 복잡한 검색 없음

DB 는 처음 세팅 부담이 크다. 파일 한 덩이로 가면:
- 들여다보기 쉽다 (텍스트 에디터로 열면 끝)
- 버전관리·백업이 단순 (cp / git 으로 충분)
- 파일 옮기는 것만으로 게임을 다른 곳으로 이동 가능

쓰기를 안전하게 하려면 임시 파일에 다 쓴 뒤 한 번에 바꿔치기 (`.tmp` → `os.replace`). 도중에 전원이 끊겨도 반쯤 쓰다 만 파일이 남지 않는다. 동시 접근은 `asyncio.Lock` (한 작업이 끝날 때까지 다른 작업이 기다리게 막는 자물쇠) 하나면 충분. 사용자 수·세션 수가 늘면 P3 이후 SQLite (가벼운 파일형 DB) 로 옮길지 검토.

### 3.12 왜 난이도 판정에는 지난 흐름을 안 주나?

난이도 판정은 "지금 이 입력이 어떤 종류 행동인가, 얼마나 어려운가" 만 분류하면 된다. 지난 턴 요약·기억·세계관까지 넣으면:
- 토큰 (LLM 이 한 번에 처리하는 글자 단위) 낭비 — 판정은 가볍고 자주 부른다
- 판정이 이야기 쪽으로 오염됨 ("어제 배신당했으니 이번 설득은 어려움" 같은 추론은 이야기 쪽 일)
- 같은 입력·같은 장면에서 결과가 달라짐 (재현 안 됨)

긴 흐름의 영향은 이미 **지금 상태**에 쌓여 있다. NPC 호감도가 떨어졌거나 성향이 경계 쪽으로 움직였다면 "주변" 정보의 상태 태그에 그대로 보인다. 난이도 판정은 그 한 장면만 보면 충분.

이야기는 반대다. "왜 지금 이런 반응인가" 를 풀어 써야 하니 지난 턴·기억이 필요하다.

→ 레이어 분리: **난이도 판정 = 지금 장면 정보만 / 이야기 = 모든 정보 (4개 레이어 전부)**.

### 3.13 왜 프론트로 보내는 데이터를 한 곳에서만 만드나?

프론트 타입 (`frontend/types/domain.ts`) 은 **UI 에 보이는 필드만** 담는다. 안쪽 도메인은 `disposition` (성향), `tone_hint` (말투 힌트), `location_id`, `memories` (기억) 같은 힌트·계산용 필드를 많이 갖고 있다. 이게 마구 프론트로 새 나가면:
- 프론트가 안쪽 구조에 묶여서 바꾸기 어려워짐
- 플레이어가 브라우저 개발자 도구로 "NPC 의 속마음 점수" 같은 걸 볼 수 있음 (게임 경험 망가짐)

→ `mapping/to_front.py` 한 곳에서만 프론트로 보낼 데이터를 만든다. 다른 어떤 길로도 안쪽 필드가 SSE/REST 응답에 실리면 안 된다. 이건 테스트로 강제 (`test_internal_fields_not_leaked`).

→ [04-boundary.md](./04-boundary.md) §1 (외부 노출 / 안쪽 전용 분리)

### 3.14 왜 모순된 기억을 덮어쓰지 않나?

"신뢰함" → "배신당함" 같은 모순 기억이 생길 때, 새 기억이 옛 기억을 지우지 않고 둘 다 남긴다. 시간순으로 풀면 "한때 신뢰했으나 배신당했다" 라는 자연스러운 서사가 나온다 — 덮어쓰면 NPC 가 지금처럼 반응하는 *이유* 가 사라진다.

### 3.15 왜 전투를 한 방 시네마틱으로 바꿨나?

처음엔 D&D 스타일의 라운드 루프로 잡았다 — 이니셔티브 굴림으로 행동 순서를 정하고, 매 라운드마다 각 actor 가 한 번씩 명중·데미지를 굴리고, 끝날 때까지 반복. 룰의 정확도는 가장 높지만 한 전투가 너무 늘어졌다. 골목길 잡몹 한 무리도 6–8 라운드씩 가니 "이야기가 흐른다" 는 감각이 끊겼다.

한 방으로 압축하면:
- 한 d20 + 스탯 vs DC 로 한 번만 굴린다. 등급 (critical_success ~ critical_failure) 이 기계적 결과 (몇 명이 죽고 / 플레이어가 얼마나 다치고 / XP 가 얼마나 들어오는지) 를 결정.
- `combat_narrate` 에이전트가 그 등급에 맞는 톤으로 5–10 문장 짜리 시네마틱 한 덩어리를 풀어 쓴다 ("검이 밤공기를 가르고...").
- 데미지 분포·이니셔티브 같은 라운드 단위 정밀도는 포기. 대신 페이스를 살린다.

기계적 정확도보다 흐름이 중요하다는 결정. 굴림 시점은 한 번 (`pending_check kind="combat_roll"` → 프론트 주사위 버튼 → `/roll` 한 번) 이라, 옛 §3.10 의 두 단계 턴 (`pending_check`) 모델은 그대로 유지된다 — 굴리는 횟수만 줄어듦.

### 3.16 왜 메타 액션 REST 엔드포인트를 없앴나?

처음엔 게임 안 모든 행동에 자기 URL 을 줬다: `POST /equip`, `POST /buy`, `POST /level-up`, `POST /cast`, `POST /use` 등. UI 가 버튼이나 폼을 띄우고 그걸 누르면 백엔드가 처리하는 흐름.

문제: 이렇게 하면 **플레이어가 룰을 알아야 뭘 할 수 있다**. "이 검은 어느 손에 들어야 하지", "이 포션은 어떤 슬롯에서 쓰지", "흥정은 어떤 화면에서 하지" — 게임이 LLM 으로 자연어를 처리하는 마당에 일부 행동만 GUI 로 뺀 게 일관성 없는 경험을 만들었다. 자유도도 줄었다 — 정해진 버튼만 누를 수 있으니까.

해결: 모든 행동을 `/turn` 의 자연어 한 통로로 모았다. 플레이어가 "철검을 든다", "약초를 먹는다", "여관 주인에게 사과 5개 사겠다" 라고 적으면 judge 가 입력을 보고 `equip` / `use` / `buy` 액션으로 분류한다. judge 의 액션 종류가 `pass / roll / combat / rest` 같은 굵직한 것에서 15 종 (장비·거래·레벨업·스킬·복합 입력 포함) 으로 늘어난 건 그 결과.

남은 endpoint 는 `/turn`, `/roll`, `/intro` 셋과 GET 류 (profiles, current, state) 뿐. "복합 입력" ("약초 먹고 검 든다") 은 judge 가 `chain` 액션으로 묶어 한 턴 안에서 sequential 처리.

> 사용자에게 룰을 요구하지 않는다 — 룰은 LLM 이 알아듣고, 플레이어는 글로 적기만 한다. 이게 이 엔진의 자연어 우선 원칙.

## 부록: 일부러 안 만들기로 한 것

구현하지 않기로 결정한 영역:

- 비전투 NPC 가 알아서 일정대로 움직이게 하기 — 매 NPC 매 턴 LLM 호출의 비용·일관성 문제 (§3.8 과 같은 논리)
- 아이템 닳음 / 부서짐
- 날씨·환경의 자동 이벤트 — §2.1 의 "약한 약속" 방침. 엔진이 알아서 바꾸지 않고 내러티브가 판단
- 파티 (파티원 골라서 데리고 다님) 시스템 — [03-features.md](./03-features.md) §2.9 의 동반자 (companions) 가 같은 요구를 포켓몬 잡듯 모으는 방식으로 충족
- 스킬 시전 중 사정거리 밖이면 자동으로 자리 옮기기 — 한 턴에 행동 하나 (§3.4) 위반, 숨은 두 번째 행동이 됨
- NPC 끼리의 거래
