# 게임플레이

이 파일은 실제 플레이 규칙을 설명한다.

## 제일 중요한 원칙

플레이어가 고른다.
engine이 판정한다.
LLM이 결과를 이야기로 말한다.

## 전투

전투는 임시 하트로 해결한다.

- 전투 시작 시 플레이어 하트 3, 적 하트 3을 둔다.
- 플레이어 입력 하나가 전투 한 번의 교환이다.
- 플레이어만 d20을 굴린다.
- 성공하면 행동별 성공 효과를 적용한다.
- 실패하면 행동별 실패 효과를 적용한다.
- 적 하트가 0이면 승리한다.
- 플레이어 하트가 0이면 패배하고, 남은 적 하트만큼 실제 HP를 잃는다.
- 4번째 교환 강제 종료 규칙은 쓰지 않는다.
- LLM은 피해량, 사망, 승패를 정하지 않는다.

## 새 전투 시작

새 전투를 여는 공격은 먼저 확인창을 띄운다.

확인 전에는 아래가 바뀌지 않는다.

- 적대 상태
- 시간
- HP/MP
- 전투 상태
- 로그

이미 전투 중이면 매 공격마다 확인창을 띄우지 않는다.

## CombatState

`CombatState`는 전투가 다음 입력까지 이어지도록 저장하는 임시 진행 상태다. 전투의 대상, 위치, 참여자는 graph node와 edge를 참조한다.

| 필드 | 쉬운 뜻 |
|---|---|
| `location` | 전투 장소 |
| `participants` | 전투에 들어간 캐릭터 |
| `sides` | 누가 누구 편인지 |
| `active_enemy_id` | 현재 상대하는 적 |
| `player_hearts` | 플레이어의 임시 전투 하트 |
| `enemy_hearts` | 적의 임시 전투 하트 |
| `round` | 몇 번째 교환인지 |
| `last_action` | 직전 플레이어 행동 |
| `last_support_id` | 직전 행동에 붙인 스킬이나 아이템 |
| `last_roll`, `last_dc` | 직전 d20 결과와 목표 난이도 |
| `trace` | 이미 일어난 전투 사건 |
| `outcome` | 진행 중인지, 승리인지, 도망인지, 패배인지 |

전투에서 가능한 행동:

- 공격
- 방어
- 도주
- 설득 같은 사회 행동
- 스킬이나 아이템을 하나 붙이는 보조 행동

스킬과 아이템은 행동을 대체하지 않고 한 번의 판정에 붙는 보조다. 한 교환에는 하나만 붙일 수 있다.

## 전투 한 번의 교환

```text
대상과 전투 가능 여부 검사
  -> 보조 스킬/아이템 검사와 비용 처리
  -> 플레이어 d20 판정
  -> 성공 또는 실패 하트 효과 적용
  -> 전투가 끝났는지 확인
  -> 승리면 적 status/defeat_mode 갱신
  -> 패배면 남은 적 하트만큼 플레이어 HP 차감
  -> LLM이 전투 장면을 씀
  -> 화면에 상태 보냄
```

LLM은 이미 정해진 결과만 묘사한다. HP 숫자, 피해량, 보상 숫자는 받지 않는다. 전투 문맥에는 하트와 상태 말만 들어간다.

## 패배와 처치

`character_defeat`는 반드시 죽었다는 뜻이 아니다.

| defeat mode | 쉬운 뜻 |
|---|---|
| `dead` | 사망 |
| `unconscious` | 쓰러졌지만 살아 있음 |
| `surrendered` | 항복 |
| `escaped` | 도망 |

퀘스트는 어떤 defeat mode를 허용하는지 적어야 한다. “죽이지 말고 제압” 퀘스트라면 `dead`는 실패일 수 있다.

## 전투 보상

전투 보상은 두 종류다.

| 종류 | 처리 |
|---|---|
| loot | 쓰러진 대상의 전리품 |
| quest reward | 퀘스트 완료 보상 |

몬스터를 자동 생성해도 LLM이 전투 수치를 만들지 않는다. engine이 템플릿으로 만든다.

## 성장

성장은 자동으로 적용되지 않는다. 플레이어가 선택해야 한다.

| 필드 | 쉬운 뜻 |
|---|---|
| `level` | 성장 단계 |
| `exp` | 경험 자원 |
| `level_cap` | 최대 레벨. 현재 10 |
| `stats` | 4개 stat |
| `max_hp`, `max_mp` | 최대 HP/MP. 1레벨 5, 최대 10 |
| `knows_skill` edge | 배운 기술. 최대 3개, 각 기술 tier 3까지 |
| `pending_growth` | 고를 수 있는 성장 선택지 |

경험치는 engine 보상으로만 오른다. LLM이 경험치를 주지 않는다.

## Stat

| stat | 쓰는 상황 |
|---|---|
| `body` | 힘, 버티기, 근접 공격 |
| `agility` | 회피, 은신, 도주, 추격 |
| `mind` | 관찰, 지식, 집중, 마법 제어 |
| `presence` | 설득, 위협, 의지 |

스탯은 전투 DC에 직접 들어가지 않는다. 스탯은 비전투 판정, 캐릭터 성향, 개인화 스킬 후보 생성에 사용한다.

성장 선택지는 최대 HP +1, 최대 MP +1, 새 기술 습득, 기존 기술 강화 중 하나다. 새 기술을 배우려면 스킬 슬롯이 남아 있어야 하고, 기존 기술은 tier 3까지만 강화할 수 있다.

6-stat처럼 세세하게 나누지 않는다.

## HP/MP 상태 말

HP/MP는 character node의 속성으로 저장한다. 하지만 LLM과 화면에는 상태 말도 같이 준다.

| resource | 상태 | 쉬운 뜻 |
|---|---|---|
| HP | `healthy` | 멀쩡함 |
| HP | `hurt` | 다쳤지만 행동 가능 |
| HP | `critical` | 위험함 |
| HP | `downed` | 쓰러짐 |
| MP | `ready` | 기술 쓸 여유가 있음 |
| MP | `strained` | 부담이 큼 |
| MP | `drained` | 거의 비었음 |

LLM은 숫자보다 상태 말을 우선 본다.

## 레벨업 흐름

```text
보상 적용
  -> engine이 성장 가능 여부 확인
  -> 성장 선택지 만들기
  -> 플레이어가 선택
  -> engine이 선택한 성장 적용
```

## 기술

`Skill`은 캐릭터가 배워서 쓰는 정해진 효과다.

| 필드 | 쉬운 뜻 |
|---|---|
| `id` | 기술을 찾기 위한 고정 이름 |
| `name`, `description` | 표시 이름과 설명 |
| `kind` | 지원, 회복, 강화, 유틸, 사회 행동 |
| `action` | 전투에서 보조할 행동. 예: attack, defend, flee, social |
| `requirements` | 배우거나 쓰기 위한 조건 |
| `mp_cost` | 스킬 사용에 드는 MP |
| `target_rule` | 누구에게 쓸 수 있는지 |
| `effect_template` | 실제 효과 템플릿. 예: DC 감소, 추가 하트 피해, 하트 손실 방지 |
| `support_bonus` | DC 감소형 효과에 쓰는 보정값 |
| `tags` | 분류용 꼬리표 |

기술 설명문은 분위기 설명이다. 전투 효과와 비용은 `effect_template`, `support_bonus`, `mp_cost`에 있어야 한다.

## 기술 후보

레벨업 때 LLM은 기술 후보를 제안할 수 있다.

LLM이 채울 수 있는 것:

- 기술 이름
- 짧은 설명
- 분위기 tag
- 역할 후보

LLM이 만들 수 없는 것:

- 피해량
- 회복량
- MP 비용
- 재사용 대기
- 대상 수

engine은 후보를 실제 효과 템플릿에 맞춘다. 맞출 수 없으면 버린다.

## 기술 사용

기술을 쓰려면 아래를 통과해야 한다.

| 검사 | 뜻 |
|---|---|
| 배웠는가 | 캐릭터와 기술 사이에 `knows_skill` edge가 있는지 |
| 조건 | level, stat, 종족, 선행 기술 조건 |
| tier | 해당 기술의 강화 단계가 1-3 범위인지 |
| 비용 | MP를 낼 수 있는지 |
| 행동 | 기술의 `action`이 현재 전투 행동과 맞는지 |
| 대상 | 대상이 보이고 규칙에 맞는지 |

전투 기술은 별도 `cast` 결과가 아니라 공격, 방어, 도주, 사회 행동에 붙는 보조다. 회복, 강화, 유틸 기술은 즉시 적용되거나 비전투 판정으로 간다.

## 시간

시간은 engine이 관리한다.

LLM은 “밤이 되었다”라고 말할 수 있다. 하지만 시간 값이나 회복량을 직접 바꾸지 않는다.

| 필드 | 쉬운 뜻 |
|---|---|
| `turn_count` | 상태가 바뀐 턴 수 |
| `day_phase` | 새벽, 오전, 오후, 밤 같은 표시 |
| `scene_id` | 같은 장면 묶음 |
| `active_effects` | 시간이 지나면 줄어드는 효과 |

분 단위 시계는 두지 않는다.

## 휴식

휴식은 회복과 시간 흐름을 함께 처리한다.

```text
휴식 요청
  -> 위험하면 확인창
  -> 장소와 위험도 검사
  -> 비용이 있으면 지불
  -> 휴식 결과 결정
  -> 회복 또는 조우
  -> 시간 흐름 처리
  -> 화면에 상태 보냄
```

| 결과 | 처리 |
|---|---|
| `safe` | 시간 흐름, HP/MP 회복 |
| `interrupted` | 일부 회복 또는 회복 없음, 조우 발생 |
| `blocked` | 조건 때문에 휴식 불가 |

전투 중에는 휴식할 수 없다. 시간이 지났다는 이야기만으로 HP/MP가 회복되지는 않는다.

## 퀘스트

퀘스트는 플레이어가 선택할 수 있는 할 일이다. 퀘스트도 graph의 node다.

| 필드 | 쉬운 뜻 |
|---|---|
| `id` | 퀘스트를 찾기 위한 고정 이름 |
| `status` | 잠김, 진행 중, 완료, 실패, 포기 |
| `title`, `hook`, `goal` | 표시 문장 |
| `scope` | 어느 범위의 퀘스트인지 |
| `triggers` | 완료 조건 |
| `fail_triggers` | 실패 조건 |
| `rewards` | 보상 |
| `edges` | 의뢰자, 목표, 아이템 보상, 위치로 이어지는 edge |

제목과 목표 문장은 안내일 뿐이다. 완료 판정은 trigger가 한다.

## 퀘스트 상태

```text
locked -> active -> completed
                 -> failed
                 -> abandoned
```

자동 생성 퀘스트는 기본적으로 `locked`에서 시작한다. 플레이어가 확인해야 `active`가 된다.

## 퀘스트 수락과 포기

퀘스트를 발견했다고 바로 시작하지 않는다. 화면에 제안으로 보여주고 확인창을 띄운다.

```text
수락 입력
  -> 확인창
  -> 확인
  -> locked에서 active로 변경
```

포기도 확인창을 띄운다. 포기는 진행 상태와 보상 가능성을 바꾸기 때문이다.

## Trigger

trigger는 engine이 확인할 수 있는 사건이어야 한다.

| trigger | 예 |
|---|---|
| `location_enter` | 특정 장소 도착 |
| `character_defeat` | 특정 캐릭터 제압 |
| `item_obtained` | 특정 아이템 획득 |
| `item_use` | 특정 아이템 사용 |
| `social_check` | 특정 NPC 설득이나 위협 성공 |

“진실을 밝혀라”처럼 확인할 방법이 없는 목표는 trigger가 될 수 없다. 그런 문장은 hook으로만 쓴다.

## 보상과 연결

보상은 퀘스트 완료 처리 안에서만 지급한다.

| reward | 처리 |
|---|---|
| `gold` | 돈 추가 |
| `exp` | 경험 자원 추가 |
| `item` | 아이템 지급 |
| `affinity` | 관계 변화 |

퀘스트는 관련 대상과 edge로 연결되어야 한다.

| edge | 뜻 |
|---|---|
| `gives_quest` | 퀘스트를 주는 대상 |
| `target_of` | 퀘스트 목표 |
| `required_by` | 필요한 아이템이나 단서 |
| `reward_of` | 아이템 보상 |
| `located_at` | 새 대상이 놓인 위치 |

돈, 경험치, 관계 변화 보상은 quest property에 둔다. `reward_of` edge는 아이템 보상에만 쓴다.

edge가 없으면 engine은 퀘스트를 소개하거나 완료를 판정하기 어렵다.

## 자동 퀘스트 생성

자동 퀘스트 생성은 플레이어가 할 일을 잃지 않게 하기 위한 기능이다.

LLM은 퀘스트를 바로 게임에 넣지 않는다. 먼저 초안을 만들고, engine이 검사한다.

```text
새 할 일이 필요함
  -> engine이 퀘스트 틀을 고름
  -> LLM이 제목과 의뢰 문장을 씀
  -> engine이 필요한 NPC/아이템/몬스터 칸을 계산
  -> LLM이 각 칸의 표현을 채움
  -> engine이 전체 묶음을 검사
  -> 통과하면 GraphChange로 graph에 넣음
  -> 소문, 의뢰, 사건으로 소개
```

## 언제 자동 생성하나

| 상황 | 생성 가능 |
|---|---|
| 진행 중인 퀘스트가 없음 | 가능 |
| 현재 장소나 NPC가 줄 퀘스트가 없음 | 가능 |
| 플레이어가 할 일을 물음 | 가능 |
| 휴식이나 장면 전환 뒤 사건이 필요함 | 가능 |
| 할 일이 충분함 | 생성하지 않음 |

## 퀘스트 묶음 초안

퀘스트 node는 혼자 있으면 안 된다. 목표, 보상, 장소, 관련 NPC가 edge로 같이 있어야 한다.

| 필드 | 쉬운 뜻 |
|---|---|
| `template` | engine이 고른 퀘스트 틀 |
| `quest` | 제목, 의뢰 문장, 목표 |
| `characters` | 의뢰자, 목표 NPC, 몬스터, 목격자 |
| `items` | 단서, 보상, 열쇠, 회수 대상 |
| `edges` | 퀘스트와 대상들의 edge |
| `triggers` | 완료/실패 조건 |
| `reward_budget` | 보상 한도 |

## 자동 퀘스트 검사

engine은 게임에 넣기 전에 아래를 검사한다.

- id가 겹치지 않는다.
- 모든 연결의 양쪽 대상이 실제로 있다.
- trigger 대상이 실제로 있다.
- 새 캐릭터와 아이템이 실제 location node나 소유 edge에 연결된다.
- 처치 목표는 전투 정보가 있다.
- 보상이 한도를 넘지 않는다.
- 목표는 engine이 확인할 수 있다.
- 새 퀘스트가 자동으로 active가 되지 않는다.

하나라도 실패하면 퀘스트 묶음 전체를 버린다.
