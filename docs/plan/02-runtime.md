# 런타임

이 파일은 플레이어 입력 하나가 어떤 순서로 처리되는지 설명한다.

## 제일 중요한 원칙

flow는 순서를 맡는다.
engine은 `GraphChange` 생성과 적용을 맡는다.
LLM은 플레이어 말뜻 고르기와 이야기 쓰기를 맡는다.
Python은 LLM이 고른 값을 검사하고 게임 행동 JSON을 만든다.

## 전체 흐름

```text
저장된 graph와 진행 상태 읽기
  -> pending_roll / pending_confirmation 확인
  -> LLM에게 보여줄 후보 만들기
  -> LLM이 의도와 대상 고르기
  -> Python이 게임 행동 JSON 만들기
  -> engine이 규칙 검사
  -> 바로 처리 / 행동 불가 / query / roll 필요 / 확인 필요로 나누기
  -> 결과를 저장하고 화면에 먼저 보내기
  -> LLM이 결과 이야기 쓰기
  -> 이야기 저장하고 화면에 보내기
```

## 후보 목록

`context`는 LLM에게 보여주는 후보 목록이다.

들어갈 수 있는 것:

- 현재 장소
- 보이는 NPC와 아이템
- 갈 수 있는 출구
- 최근 대화와 로그
- 상인이 파는 물건
- 플레이어 인벤토리
- 플레이어 기술

`context`는 server가 저장된 graph에서 만든다. LLM은 context를 만들지 않는다.

`context`는 참고 자료일 뿐이다. engine이 믿는 사실은 저장된 `graph`와 engine 검사를 통과한 변경뿐이다.

## 의도 고르기

이 단계는 플레이어 말이 무슨 행동인지 고른다.

LLM이 고르는 것:

- intent
- `context` 안에 있는 대상 id

예:

```json
{
  "intent": "buy",
  "merchant_id": "merchant_01",
  "item_id": "potion_01"
}
```

LLM이 하면 안 되는 일:

- context 만들기
- 최종 게임 행동 JSON 만들기
- 성공/실패 정하기
- 난이도 정하기
- graph 바꾸기
- 없는 대상을 새로 만들기

## 게임 행동 만들기

Python은 LLM이 고른 값을 검사하고 최종 게임 행동 JSON을 만든다.

검사하는 것:

- intent가 허용된 행동인지
- id가 `context` 후보 안에 있는지
- 필요한 값이 빠지지 않았는지
- 여러 행동이 섞인 입력이면 안전하게 순서대로 나눌 수 있는지

후보에 없는 id는 믿지 않는다. 값이 부족하거나 여러 행동을 안전하게 나눌 수 없으면 질문으로 돌리거나 거절한다.

## Engine 결과 나누기

engine은 Python이 만든 게임 행동 JSON을 검사하고 다음 길을 고른다.

| 길 | 쓰는 경우 | 결과 |
|---|---|---|
| 바로 처리 | 지금 실행 가능 | graph와 진행 상태 변경 |
| 행동 불가 | 조건이 맞지 않음 | 실패 결과 정리 |
| query | 질문이나 정보 확인 | 공개 정보만 읽어 답함 |
| roll 필요 | 주사위가 필요함 | roll 대기 저장 |
| 확인 필요 | 사용자의 확인이 필요함 | 확인 대기 저장 |

`query`는 여기서 끝난다. `query`는 graph, 시간, 대기 상태, 이야기를 바꾸지 않는다.

전투 중이면 `query`를 뺀 대부분의 행동은 전투 교환으로 처리한다.

## 대기 상태

pending은 “지금 다른 입력을 받기 전에 먼저 끝내야 하는 일”이다.

한 번에 하나만 있을 수 있다.

| pending | 플레이어가 해야 할 일 |
|---|---|
| `pending_confirmation` | 확인 또는 취소 |
| `pending_roll` | 주사위 굴림 |

`pending_confirmation`이 있으면 일반 입력과 주사위 입력을 막는다.
`pending_roll`이 있으면 일반 입력과 확인 입력을 막는다.

## 확인

confirmation은 확인창이다.

```text
확인이 필요한 게임 행동 JSON
  -> 원래 게임 행동 JSON 저장
  -> 확인창을 화면에 보냄
  -> 멈춤

확인
  -> 저장했던 게임 행동 JSON을 다시 처리

취소
  -> 확인 대기만 제거
```

확인은 성공이 아니다. 확인 뒤에도 조건이 안 맞으면 행동은 실패하거나 거절될 수 있다.

## 주사위

roll은 주사위 판정 대기다.

flow는 원래 게임 행동 JSON을 저장하고 멈춘다. 플레이어가 주사위를 굴리면 저장된 행동을 불러와 engine에 넣는다.

roll 결과를 받은 뒤에는 다시 의도를 고르지 않는다.

복잡한 행동에 판정이 여러 개 필요하면 한 번에 다 굴리지 않는다. 첫 판정 뒤 다음 행동은 다음 입력으로 넘긴다.

## Engine 적용

engine은 현재 graph에서 그 행동이 가능한지 검사하고 결과를 만든다.

```text
검사
  -> 안 되면 거절
  -> 되면 GraphChange 만들기
  -> GraphChange 검사 규칙 확인
  -> graph에 적용
  -> 저장할 변경 모으기
  -> 이어질 일 모으기
```

저장해야 할 변경은 `Dirty`, 저장 뒤 이어서 처리할 일은 `Effect`라고 부른다.

engine 결과는 먼저 저장하고 화면에 보낸다. 이야기는 그 다음에 쓴다.

## 후속 처리

`Effect`는 어떤 일이 끝난 뒤 자연스럽게 이어지는 일이다.

| 순서 | Effect | 쉬운 뜻 |
|---|---|---|
| 1 | character defeat / flee | 전투 결과, 처치 상태, loot 정리 |
| 2 | quest completed / failed | 퀘스트 완료/실패와 보상 처리 |
| 3 | level ready | 성장 선택지 열기 |
| 4 | sleep encounter needed | 휴식 중 조우 처리 |
| 5 | quest needed | 할 일이 부족하면 퀘스트 초안 만들기 |

전투 결과가 있으면 일반 이야기 호출 대신 `combat_narrate`를 쓴다.

## 새 콘텐츠 넣기

새 아이템, 캐릭터, 퀘스트는 바로 게임에 넣지 않는다.

```text
새 내용이 필요함
  -> engine이 템플릿과 선택지를 정함
  -> LLM이 짧은 초안을 씀
  -> engine이 전체를 검사함
  -> 통과하면 GraphChange로 graph에 넣음
  -> 실패하면 버림
```

퀘스트 초안은 특히 한 묶음으로 검사한다. 퀘스트, NPC, 몬스터, 아이템, edge, trigger 중 하나라도 이상하면 전체를 버린다.

## 이야기 쓰기

이야기 쓰기는 이미 결정된 결과를 문장으로 바꾼다.

LLM은 여기서 성공/실패를 다시 정하지 않는다. graph도 바꾸지 않는다.

만들 수 있는 것:

| 출력 | 쉬운 뜻 |
|---|---|
| `turn_summary` | 이번 턴 요약 |
| `memory` | 캐릭터가 기억할 내용 |
| `suggestions` | 화면에 보여줄 행동 제안 |

## 저장과 화면 출력

flow는 변경된 graph와 진행 상태를 저장한다.

화면 출력은 두 번 나갈 수 있다.

1. 결과 먼저 보내기
2. 이야기 나중에 보내기

화면에 보낼 문장과 label은 server가 만든다. client는 받은 내용을 그대로 보여준다. client가 날짜, 상태 이름, 버튼 문구를 새로 조립하지 않는다.
