## 미해결 문제

### A. Quest giver 학살 (가장 심각)

**증상**: 플레이어가 `들쥐` / `광장 경비병` 같은 시드 외 적을 호명. judge 가 combat 으로 분류한 뒤 target 골라야 하는데, location 에 alive NPC 가 의뢰자 (거륜) 한 명뿐 → fallback rule (b) 발동 → 거륜이 맞아 죽음. 의뢰자 사망 시 quest 진행 차단.

**재현**: `combatant` agent 에서 두 라운드 연속 같은 패턴.

**Fix 방향**:
- `dc_judge/prompt.md` 의 Combat target rule (b) 에 hostility 검사 추가 — friendly NPC (의뢰자·상인) 는 fallback 에서 제외, 곧장 (c) summon_combat 으로
- 또는 `dc_judge/semantics.py:_check_combat` 에 "friendly target → reject + retry" 로직 (다만 substring 휴리스틱이라 robust 하지 않음)
- 가장 깔끔: 코드층에서 friendly NPC 를 combat target 으로 받으면 자동 거부하고 narrate 가 *"검을 들었지만 그가 친구임을 깨닫는다"* 로 흡수. LLM 호출 없이 결정론적.

### B. 일부 agent 가 여전히 광장에서 못 나감

**증상**: narrate move auto-inject fix 덕에 일부는 이동하지만 (questor → outpost_tower 도달), 7-8 명은 여전히 광장에 머묾. judge 가 `"여관으로 간다"` 류 입력에서 `pass targets=[connection_id]` 로 잡는 비율이 낮음.

**원인 분석 필요**: QA agent personality 가 대화 위주 (이게 더 가능성 높음) vs judge prompt 의 movement 룰이 약함.

**Fix 방향 (옵션)**:
- (휴리스틱) judge 결과 후 player_input 에 connection name substring 매치 → 자동 inject. 여관이 여러 개 같은 케이스에서 부서짐. 권장 안 함.
- (구조적) `MoveAction(destination: str)` 을 judge 스키마의 first-class action 으로 추가. 이동 의도면 schema validation 으로 destination 강제. 1-2시간 작업.
- (그냥 둠) narrate 가 *"발걸음을 옮기지만 망설인다"* 로 흡수, 다음 턴에 플레이어가 명시 — 코드 변경 0.

### C. LLM 단일 게임 분산

**증상**: prompt 단속 추가에도 단일 게임 변동 큼.
- `survivor` 축축/눅눅 6→**10** (역행)
- `griefer` 메타 동사 4→**5** (역행)
- 동일 prompt·동일 모델인데 매 게임 다르게 나옴.

**Fix 방향**:
- (코드층 후처리) post-narrate 자동 검증 — 직전 N 턴의 본문에서 키워드 카운트 추적, 임계 (예: `짙은 안개` 3회) 초과면 narrate retry. narrate 의 streaming 특성 때문에 mid-stream retry 는 못 하므로 finalize 직전에 검사하고 다음 턴부터 강제 disabled-vocab 리스트로 prompt 보강.
- (시드 단계) `world.md` 의 분위기 키워드 다양화 — LLM 이 시드 vocab 을 무의식적으로 끌어옴. `"짙은 안개"` 외 `"낮은 하늘"`, `"옅은 빛"`, `"습한 바람"` 같은 변주 박아두기.

### D. 시드 외 entity 환각

**증상**: 플레이어가 *"지하 던전 가겠다"* 같은 시드에 없는 location 호명 → narrate 가 *"어둡고 축축한 지하 통로"* 를 발명. narrate prompt 의 "engine-tracked entity 발명 금지" 룰 위반.

**예**: combatant T10·survivor T13.

**Fix 방향**:
- (코드층 후처리) narrate 본문 finalize 시 `state.locations` 외 location 명을 검출하면 retry. 또는 본문 단어가 `surroundings.entities` + `state.items` + `state.locations` 의 known 명단에 한정되도록 검사. 정밀 NER 어려우니 휴리스틱.
- (prompt 강화) 이미 룰에 있음 — LLM 이 어김. 더 강한 enforcement 필요.

### E. Grade table 위반

**증상**: `griefer` T2 — CHA roll fail (3/9) 인데도 거륜이 quest 정보 전부 줌. narrate prompt 의 grade 표 ("failure → NPC가 결국 사실 흘려주는 우회 성공 금지") 위반.

**Fix 방향**:
- prompt 강화 (이미 룰 있음, 어김)
- 또는 grade=failure 시 narrate body 길이 / NPC 발화량 cap (코드층 검사 — 정량 측정 어려움)
- 가장 실용적: post-narrate finalize 시 `「...」` 직접 인용 길이가 일정 이상이면 grade=failure 케이스에서 retry

### F. NPC 톤 차별 검증 부족

**증상**: 단일 NPC (`거륜`) 만 등장하는 게임이 압도적. 마야·주옥·라크·타름이 같이 나오는 turn 이 없어서 톤 차별 효과를 transcript 로 검증 불가.

**Fix 방향**:
- QA 시나리오를 NPC 다수 등장하는 구조로 다시 짜기 (마을 광장에 여러 NPC 동시 배치)
- 또는 QA agent personality 를 이동 적극형으로 (현재는 대화 위주) — 마야·주옥 자연스럽게 만나게

## 나레이트 에이전트 수정 방향 (정리)

prompt 단속만으로는 한계가 명확. 다음 라운드는 **코드층 후처리** 가 본격적으로 필요한 영역:

### 1. post-narrate 검증 + 다음 턴 강제 disable list

narrate finalize 직전:
- 직전 N 턴 본문의 키워드 N-gram 카운트
- 임계 초과 키워드 → 다음 턴 prompt 에 `forbidden_vocab: [...]` 동적 inject
- streaming 특성상 mid-stream retry 는 안 되니 다음 턴 enforcement 로

### 2. seed entity 명단 vs narrate body 검증

narrate 본문에서 surroundings/state 외의 entity 명사 검출 → retry. 정밀 NER 까지 안 가도 "시드에 없는 location/character/item 명" 검사로도 충분.

### 3. grade-aware body cap

grade=failure 시 NPC 직접 인용 길이 또는 quest 단서 노출 검사. 이건 정량화 어려운 영역이지만 tail-end 휴리스틱 (NPC 인용 200자 초과 + grade=failure → retry) 정도는 가능.

### 4. failure-path narrate routing (옵션 A)

엔진 reject (use 실패 등) 가 narrate 거치지 않고 즉시 메시지 뱉는 현재 구조 → narrate pass 로 흡수. push_act 으로 분리 했으니 시각은 OK 지만, *"몸이 차 있어 약초를 내려놓는다"* 같은 진짜 prose 가 좋아 보이는 케이스. LLM 호출 비용 추가.

### 5. world.md 키워드 다양화

시드 단계에서 narrate 가 끌어오는 vocab 풀 확장. `"짙은 안개"` 만 박지 말고 5-6 가지 분위기 표현 박아두기. 어휘 다양성의 시드 측 해법.

### 6. judge Move first-class action

이동 의도를 `MoveAction(destination)` 로 schema 화. movement 가 `pass targets=[loc_id]` 와 dialogue 호명 사이에서 헷갈리는 현재 구조 정리.
