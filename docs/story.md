# 시나리오 원천 문서: 안개와 흰섬

이 문서는 게임 시나리오 seed를 만들기 위한 원천 문서입니다. 산문 감상용이 아니라 `chapters`, `locations`, `characters`, `quests`, `items`, `start` 파일로 분해하기 쉽도록 정리합니다.

## 시나리오 목표

- 플레이어는 안개섬에서 엘리와 동행을 시작하고, 감정과 사회 문제가 결합된 섬들을 순서대로 지나갑니다.
- 각 감정 섬은 하나의 핵심 사건, 네 명 안팎의 주요 NPC, 하나의 최종 판단을 가집니다.
- 최종 판단은 검은 선택과 흰 선택 두 가지로만 둡니다. 검은 선택은 검은 보상 아이템을 남기고, 흰 선택은 공식 물건을 남기지 않습니다.
- 흰 선택은 실패나 손해가 아니라 공식 물건을 남기지 않는 처리입니다. 다음 섬 진행은 검은 선택과 동일하게 열립니다.
- 마지막 흰섬은 검은 아이템 보유 여부에 따라 엘리 동행 엔딩과 엘리 부재 엔딩으로 갈라집니다.

### 섬 간 이동 규칙

- 섬 사이는 위 표의 순서대로만 진행합니다.
- 이전 섬으로 돌아가는 연결은 만들지 않습니다. 예: 붉은섬에 도착한 뒤에는 안개섬으로 돌아갈 수 없습니다.
- seed의 `location.connections`는 같은 섬 내부 자유 이동에 우선 사용합니다.
- 다음 섬으로 넘어간 뒤에는 이전 섬의 NPC, 장소, 아이템을 현재 장면에 다시 배치하지 않습니다. 필요한 경우 `turn_summary`나 공개 기록으로만 회상합니다.

### Location 연결 원칙

- 각 섬은 현재 문서의 2-3개 location 구조를 유지합니다.
- 같은 섬 내부 location은 기본적으로 양방향 연결합니다.
- 각 섬의 시작 location은 이전 섬에서 도착하는 항구, 광장, 입구 역할 location입니다.
- 다음 섬으로 가는 출구는 해당 섬의 최종 판단 완료 후에만 열립니다.
- 다음 섬 연결은 해당 섬의 최종 판단 location 또는 출항 location에서만 둡니다.

### 다음 섬 출구 seed 변환 주의

- raw `location.connections`는 정적 `connects_to` edge가 되므로, 다음 섬 연결을 초기 seed에 넣으면 조기 이동이 가능해집니다.
- 새 `white_isle` seed에서는 다음 섬 출구 connection에 `requires_quest`를 둡니다.
- `requires_quest`는 해당 quest가 completed일 때만 그 출구 이동을 허용하는 runtime gate입니다.
- 섬별 `연결 메모`의 다음 섬 출구는 초기 연결 목록이 아니라, 최종 판단 완료 후 열릴 연결 후보입니다.

### 반복 구조

- 각 섬에 도착합니다.
- 섬 안에서는 여러 장소와 NPC를 자유 순서로 탐색합니다.
- 탐색 노드는 장소, NPC, 물건, 공개 지식으로 배치합니다.
- 의뢰인, 흰색 대변인, 검은색 대변인은 서로 다른 처리 방식을 제안하지만 만나는 순서는 고정하지 않습니다.
- 엘리는 상황을 정리하고 감정의 핵심 질문을 드러냅니다.
- 플레이어는 충분히 살펴봤다고 판단하면 검은 선택 또는 흰 선택으로 최종 처리 방식을 정합니다.
- 검은 선택은 해당 섬의 검은 보상 아이템을 남기고, 흰 선택은 아이템을 남기지 않습니다. 이후 다음 섬으로 이동합니다.

### 엘리 운영 원칙

- 엘리는 정답을 알려주지 않습니다.
- 각 섬에서 엘리는 핵심 질문 1개만 던집니다.
- 판단 근거가 부족하면 부족한 확인 대상을 되묻습니다.
- 최종 판단 직전에는 검은 선택과 흰 선택의 의미를 짧게 정리합니다.
- 흰섬에서는 검은 보상 아이템 보유 여부에 따라 동행 또는 부재로 갈라집니다.

### 검은 선택 처리 규칙

- 검은 선택은 해당 섬의 검은 보상 아이템을 inventory에 추가합니다.
- 흰 선택은 아이템을 지급하지 않습니다.
- 성향 점수는 사용하지 않습니다. 엔딩 분기는 검은 보상 아이템 보유 여부로만 판단합니다.
- `black_item_count`는 현재 보유 중인 검은 보상 아이템 수입니다.
- `last_black_item`은 획득 순서가 가장 늦은 검은 보상 아이템입니다.

### 흰섬 엔딩 규칙

- `black_item_count == 0`: 엘리와 함께 흰섬에 도착합니다. 플레이어가 빈 의자에 앉으면 엘리도 옆에 앉습니다.
- `black_item_count >= 1`: 마지막 항해 중 가장 마지막에 얻은 검은 아이템이 바다로 떨어집니다. 엘리가 그것을 잡으려다 파티에서 이탈하고, 플레이어는 혼자 흰섬에 도착합니다.
- 흰섬은 아이템을 심판하지 않습니다. 붙잡은 것이 있으면 마지막 항해에서 그것이 무게가 됩니다.
- 엘리 이탈은 죽음으로 확정하지 않습니다. 엔딩에서 곁에 없게 되는 처리입니다.

## 시나리오 seed 작성 메모

- `chapter_*`는 `chapters.json` 후보 ID입니다.
- `loc_*`는 `locations.json` 후보 ID입니다.
- `npc_*`는 `characters.json` 후보 ID입니다.
- `quest_*`는 `quests.json` 후보 ID입니다.
- `item_*`는 `items.json` 후보 ID입니다.
- `knowledge_*`는 `knowledge.json` 후보 ID입니다.
- 후보 ID는 실제 seed 작성 시 기존 스키마와 중복 여부를 확인한 뒤 확정합니다.
- 각 섬의 핵심 퀘스트는 가능하면 하나로 둡니다. 탐색 노드를 필수 보조 퀘스트로 쪼개면 자유도가 줄어드므로, 기본값은 장소/NPC/아이템/공개 지식으로 표현합니다.
- 섬 내부 연결은 양방향이어도 되지만, 섬과 섬 사이 연결은 다음 섬 방향으로만 둡니다.
- 플레이어에게 보이는 한국어 문장은 2인칭 존댓말 합니다체를 유지합니다.

### Knowledge 분해 원칙

- `자유 탐색 노드`의 `알 수 있는 것`은 가능하면 `knowledge.json`의 짧은 fact로 분해합니다.
- `판단에 필요한 확인`의 근거는 새 fact를 만들기보다 자유 탐색 노드에서 얻는 fact를 참조합니다.
- 같은 사실을 NPC 대사와 장소 설명에 중복 저장하지 않습니다. NPC나 location은 관련 knowledge를 드러내는 통로로 둡니다.
- 판단을 강제하는 knowledge lock은 만들지 않습니다. 근거가 부족할 때 엘리가 되묻는 narration 지침으로 처리합니다.

### 보조 seed 파일 처리 원칙

- `actions.json`, `skills.json`, `races.json`, `slots.json`, `mbti.json`, `player.json`은 시나리오 고유 규칙이 필요해질 때까지 기존 기본 구조를 재사용합니다.
- `effects.json`은 1차 seed에서 새 효과를 추가하지 않습니다. 검은 아이템과 흰섬 분기는 우선 narration 규칙과 inventory 상태로 처리합니다.
- `factions.json`은 섬별 사회 집단을 별도 시스템으로 쓰지 않는 한 비워두거나 최소 기본값만 둡니다.
- 보조 파일에는 섬별 핵심 판단 로직을 중복 저장하지 않습니다.

### 중복 레코드 방지 원칙

- `npc_eli`는 전역 단일 character입니다. 각 섬의 Character 후보에 반복되어도 `characters.json`에는 한 번만 생성합니다.
- 검은 보상 아이템은 전역 목록과 섬별 Item 후보에 함께 나오더라도 `items.json`에는 ID별로 한 번만 생성합니다.
- 섬별 Character 후보의 반복 표기는 해당 섬에서 엘리나 아이템이 어떻게 쓰이는지 설명하기 위한 배치 메모입니다.

### Quest trigger 작성 원칙

- 사건 시작은 quest 완료 trigger로 두지 않고, 시작 location의 narration, public knowledge, visible target으로 처리합니다.
- 각 섬의 required quest는 최종 판단 trigger 1개만 완료 조건으로 둡니다.
- 감정 섬의 최종 판단 trigger는 `social_check`로 통일하고, target은 해당 섬의 의뢰인 또는 최종 판단 당사자 NPC로 둡니다.
- 최종 판단은 trigger 두 개가 아니라 quest의 `choices`로 표현합니다.
- `choices`의 key는 `record`, `release`, `open`, `restore`처럼 해당 시나리오의 행동 의미를 드러내는 일반 id로 둡니다. 서버는 이 id의 의미를 해석하지 않고 데이터로만 검증합니다.
- 검은 선택에 해당하는 choice는 검은 보상 아이템을 `choices.<choice>.rewards.items`에 넣고, 흰 선택에 해당하는 choice는 빈 보상으로 둡니다.
- 플레이어가 판단을 확정하면 server action은 `verb=decide`, `what=<quest_id>`, `how=<choice_id>`를 사용합니다.
- 최종 판단에 쓰이는 물건은 trigger target이 아니라 choice label, location, item props로 둡니다. 흰섬만 마지막 `location_enter` trigger를 사용합니다.

### Outcome seed 변환 주의

- 이 문서의 검은/흰 `outcome`은 시나리오 작성 개념입니다. seed와 server 계약에서는 `choices`와 `decide.how`로 표현합니다.
- 감정 섬 quest의 정적 `rewards.items`에는 검은 보상 아이템을 넣지 않습니다. 공통 보상으로 들어가면 어느 선택에도 같은 아이템이 지급됩니다.
- 선택별 보상은 `choices.<choice>.rewards`에만 둡니다.

### 진행 상태 작성 원칙

- `chapter_fog_island`와 `q_fog_depart`만 초기 `active`로 둡니다.
- 이후 chapter와 required quest는 초기 `locked`로 둡니다.
- 각 chapter는 바로 이전 chapter 완료를 prerequisite으로 둡니다.
- 각 섬의 required quest는 해당 chapter가 active가 된 뒤 활성화되도록 둡니다.
- 다음 섬 출구는 이전 섬 required quest 완료 후에만 의미 있게 사용됩니다. 초기 raw connection으로 바로 열어두지 않습니다.
- 흰섬의 `quest_white_empty_chair`는 마지막 required quest이며, 완료 즉시 엔딩으로 이어집니다.

### 핵심 상황

- 플레이어는 안개가 짙은 항구에 도착합니다.
- 붉은섬으로 가는 배는 출항 준비 중이지만, 안개 바다에서는 반드시 두 사람이 함께 타야 합니다.
- 정식 안내인은 안개 때문에 오지 않고, 항구 임시 조수 엘리가 대신 동행을 제안합니다.

### 핵심 상황

- 검은섬은 별도 장소로 두지 않습니다. 모든 플레이어는 마지막에 흰섬으로 향합니다.
- 검은 아이템이 없으면 엘리와 함께 흰섬에 도착합니다.
- 검은 아이템이 하나 이상 있으면 마지막 항해 중 가장 마지막에 얻은 검은 아이템이 바다로 떨어지고, 엘리가 그것을 잡으려다 파티에서 이탈합니다.
- 흰섬은 심판하지 않고, 마지막에 빈 의자 하나를 내어줍니다.
