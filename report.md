# TRPG QA Report

테스트 일시: 2026-05-11 KST  
대상: `client` web `http://localhost:8081`, server `http://localhost:8001`  
환경: 인앱 브라우저, 모바일 뷰포트 412x915, 로컬 개발 시나리오 `개발 원턴 테스트`

## 범위

- 새 게임 생성 기본값: 주인공 / 남성 / 개발 원턴 테스트 / 인간 / 한국어
- 게임 인트로, 상단 패널, 미니맵, 추천 행동, 직접 입력
- 판정/주사위, 퀘스트 수락 확인, 전투 진입/승리, 레벨업
- 새 의뢰 생성 이후 NPC/퀘스트/미니맵 상태, 퀘스트 취소/재수락
- 두 번째 전투의 도주 처리, 새 이야기 확인 모달, 영어 로케일 시작
- 콘솔 경고와 접근성 DOM 스냅샷

## 요약

현재 UI의 기본 구조와 조작 흐름은 작동합니다. 모바일 한 화면에서 상단 상태, 로그, 추천 행동, 입력창이 고정적으로 보여서 핵심 루프는 이해할 수 있습니다. 퀘스트 수락 확인 모달, 전투 패널, 레벨업 선택 UI는 각각 수락, 처리, 결과 반영까지 이어집니다.

가장 큰 문제는 LLM 서사가 엔진 이벤트와 자주 어긋난다는 점입니다. 대화 액션에 NPC 응답이 없고, 퀘스트 수락/전투 후에도 이전 "테스트 가이드에게 말을 겁니다" 장면을 반복합니다. 새로 생긴 마을 주민에게 말을 걸어도 가이드와 허수아비 장면이 반복되어, 사용자는 상태가 바뀌었다는 엔진 로그는 보지만 게임이 반응한다는 감각을 얻기 어렵습니다.

두 번째 문제는 상태 무결성입니다. 영어 로케일로 새 게임을 시작하면 첫 화면은 열리지만 첫 입력에서 `sendGraphInput failed: HTTP 404 (game not found)`가 플레이어 로그에 그대로 찍혔습니다. 전투에서 `도주`를 눌렀을 때도 내부 dispatch는 `fled`였지만 결과 로그는 승리로 표시되고 퀘스트는 `0/1`에 머물렀습니다.

세 번째 문제는 대기와 조작 피드백입니다. 첫 인트로 완료까지 약 45초가 걸렸고, 그동안 화면은 `...`만 보여줬습니다. 추천 행동도 패널이 열린 상태에서는 첫 클릭이 행동 전송이 아니라 패널 닫기로 소모되어, 누른 줄 알았는데 아무 일도 안 일어난 것처럼 보입니다.

## 주요 발견

### P0: LLM 서사가 최신 엔진 이벤트를 반영하지 못함

관찰:
- 퀘스트 수락 후 `당신은 훈련 전투 퀘스트를 시작합니다.` 엔진 로그가 찍혔지만, 다음 GM 서사는 이전 대화 액션인 `테스트 가이드에게 말을 겁니다`를 다시 반복했습니다.
- 전투 시작/승리 후에도 같은 장면 설명과 같은 대화 행동을 반복했습니다.
- 전투 승리 뒤 `새 의뢰가 도착합니다: 마을의 부탁.`은 표시되지만, GM 서사는 새 의뢰나 승리 보상보다 이전 장면을 계속 말합니다.

개선안:
- `graph_narrate` 입력에 "이번 턴의 엔진 이벤트"를 별도 우선순위 블록으로 넣고, 이전 GM 문단은 요약 또는 최근 1개 이하로 제한해야 합니다.
- 내레이션 프롬프트에 "직전 GM 문단을 재서술하지 말고, 엔진 이벤트 결과만 새롭게 묘사하라"는 명시 규칙이 필요합니다.
- 응답 후처리에서 직전 GM 문단과 높은 유사도가 감지되면 재시도시켜 반복 문제를 줄일 수 있습니다.

### P0: 대화 의도가 대화로 처리되지 않음

관찰:
- 추천 행동 `테스트 가이드에게 말을 겁니다` 결과가 `가이드는 대답하지 않고 당신을 다시 봅니다.`로 끝났습니다.
- 직접 입력 `테스트 가이드에게 허수아비 훈련 방법을 물어봅니다.`는 대화가 아니라 `지력 판정이 필요합니다`로 분기했습니다.
- NPC 패널에는 테스트 가이드가 `대화 가능 · 퀘스트 안내`로 표시되지만 실제 반응은 대화 가능 상태와 맞지 않습니다.

개선안:
- classify에서 `말을 걸다`, `물어보다`, `대화하다` 계열은 기본적으로 dialogue intent로 강하게 보내야 합니다.
- NPC가 `대화 가능` 특성을 가진 경우에는 판정보다 NPC 대사를 우선하는 규칙이 필요합니다.
- 판정이 필요하더라도 먼저 NPC의 짧은 응답을 주고, 추가 정보를 얻기 위해 판정을 요구하는 흐름이 더 자연스럽습니다.

### P0: 영어 로케일 새 게임이 첫 입력에서 404로 깨짐

관찰:
- 새 이야기 화면에서 `English`를 선택해 `Alex`로 시작하면 첫 장면은 표시됩니다.
- 첫 장면은 `You arrive at 테스트 허브...`처럼 영어와 한국어가 섞입니다.
- 이후 `Ask the guide how to defeat the training dummy.`를 보내면 로그에 `sendGraphInput failed: HTTP 404 (game not found)`가 그대로 표시됩니다.
- 이 상태에서도 클라이언트는 게임 UI를 유지하므로 플레이어는 저장이 깨졌는지, 서버 오류인지 알 수 없습니다.

개선안:
- `initGraphSession` 응답으로 받은 `game_id`가 저장되고 서버에서 즉시 조회 가능한지 클라이언트/서버 양쪽에서 검증해야 합니다.
- 404를 플레이어 로그에 원문으로 넣지 말고, 게임 복구/새로고침/새 시작 안내가 있는 오류 상태로 전환해야 합니다.
- 영어 로케일이 아직 지원 대상이 아니라면 선택지를 숨기거나 "서버 서사만 영어" 같은 제한을 명확히 해야 합니다.

### P0: 전투 도주 결과가 승리로 처리되고 퀘스트 진행과 불일치함

관찰:
- 두 번째 전투에서 `도주`를 눌렀습니다.
- 서버 로그의 `graph_narrate` query에는 `action.verb=move`, `action.how=flee`, `dispatch.outcome=fled`, `combat.trace.kind=player_fled`가 들어갔습니다.
- 같은 query의 `resolved_results`에는 `당신은 전투에서 승리합니다.`가 들어갔고, 실제 UI도 승리 로그를 표시했습니다.
- 퀘스트 패널은 이후에도 `마을의 부탁 · 0/1`이고 보상은 지급되지 않았으며, `떠돌이 적을 공격합니다` 추천 행동도 남았습니다.

개선안:
- 전투 runtime에서 `fled`와 `victory` 결과를 분리하고, `resolved_results`가 dispatch outcome을 그대로 반영하도록 테스트를 추가해야 합니다.
- 퀘스트 목표 완료 조건은 전투 승리와 같은 확정 이벤트에만 연결하고, 도주에는 유지/실패/재시도 중 하나를 명확히 적용해야 합니다.
- 전투 결과 후 추천 행동 목록에서 사라져야 할 대상과 남아야 할 대상을 검증하는 회귀 테스트가 필요합니다.

### P1: 첫 인트로 대기 시간이 길고 피드백이 부족함

관찰:
- 새 게임 클릭 후 게임 UI로 전환되는 데는 약 3초였지만, 첫 GM 문단 완료까지 추가로 약 39초, 전체 약 45초가 걸렸습니다.
- 대기 중에는 로그 위치에 `...`만 보여서 서버/LLM이 정상 처리 중인지, 얼마나 기다려야 하는지 알기 어렵습니다.
- 입력창은 `행동 중단` 상태가 되지만, 초반 사용자에게는 "멈춰도 되는지 / 지금 무엇을 기다리는지"가 불명확합니다.

개선안:
- 첫 인트로에는 단계형 상태 문구가 필요합니다. 예: `세계 불러오는 중`, `첫 장면 작성 중`, `응답이 길어지고 있습니다`.
- 10초 이상이면 짧은 안내 또는 재시도/중단 안내를 보여주는 것이 좋습니다.
- 가능하면 인트로는 짧은 템플릿 문단을 먼저 보여주고, LLM 확장 서사를 뒤이어 붙이는 방식이 체감 속도를 크게 개선할 수 있습니다.

### P1: 추천 행동 첫 클릭이 열린 패널 닫기로 소모됨

관찰:
- `미니맵` 패널이 열린 상태에서 하단 추천 행동 `테스트 가이드에게 말을 겁니다`를 클릭했을 때 행동이 전송되지 않았고 패널만 닫혔습니다.
- 같은 버튼을 패널이 닫힌 뒤 다시 눌러야 실제로 행동 전송이 됐습니다.

개선안:
- 추천 행동 버튼 클릭은 패널 상태와 무관하게 항상 행동으로 처리되어야 합니다.
- 패널 닫기는 별도 닫기 버튼, 바깥 영역 탭, 또는 같은 탭 재클릭에만 묶는 편이 안전합니다.
- 행동이 전송되지 않는 클릭이라면 시각적으로 "패널 닫힘"이 명확히 드러나야 합니다.

### P1: 판정 실패 후 서사 결과가 없음

관찰:
- 지력 판정 실패 후 롤 카드에는 `실패 · 지력 · 주사위 8 · 5 부족`만 표시됐습니다.
- 실패가 장면에 어떤 영향을 주는지, 다음 선택지가 무엇인지에 대한 GM 설명이 없습니다.

개선안:
- 판정 결과 로그 뒤에는 성공/실패 모두 짧은 결과 문단이 필요합니다.
- 실패 시에도 진행 가능한 선택지를 하나 이상 암시해야 플레이어가 막혔다고 느끼지 않습니다.

### P1: 훈련용 허수아비 전투 톤이 부자연스러움

관찰:
- `훈련용 허수아비`가 전투 시작 직후 플레이어를 공격해 HP를 27에서 20으로 낮췄습니다.
- 묘사는 `날카로운 검이 허공을 가르며 당신의 어깨를 찌릅니다.`로, 훈련용 더미보다는 살상 적처럼 느껴집니다.
- 적 HP는 `1/15`로 거의 끝난 상태였는데 플레이어가 큰 피해를 받은 이유가 명확하지 않습니다.

개선안:
- 시나리오/노드 속성에 `training`, `nonlethal`, `dummy` 같은 전투 톤 힌트를 넣고 내레이션에서 강제하는 것이 좋습니다.
- 전투 로그에 피해량, 주고받은 공격, 남은 HP 변화가 더 명확히 드러나야 합니다.

### P1: 새 NPC/적이 생겨도 패널과 지도에서 맥락을 찾기 어려움

관찰:
- 전투 승리 후 카운터는 `인물 3 · 장소 1 · 할 일 2`로 바뀝니다.
- NPC 패널을 열면 여전히 `테스트 가이드`만 보이고, 새로 생긴 `마을 주민`으로 전환하는 UI가 보이지 않았습니다.
- 미니맵은 계속 `테스트 허브`와 `준비실`만 강조하고, 새 의뢰의 마을 주민/떠돌이 적 위치를 이해시키지 못했습니다.

개선안:
- NPC 패널은 여러 인물이 있을 때 리스트/세그먼트/이전·다음 전환을 제공해야 합니다.
- 퀘스트 대상 NPC와 적은 미니맵 또는 주변 패널에서 강조되어야 합니다.
- 카운터 `인물 3`을 누르면 "늘어난 인물 1명"이 어디에 있는지 바로 확인할 수 있어야 합니다.

### P1: 퀘스트 확인 모달 취소 후 컨텍스트가 닫힘

관찰:
- `마을의 부탁` 수락 모달에서 `취소`를 누르면 모달만 닫히는 것이 아니라 퀘스트 패널도 함께 닫혔습니다.
- 같은 퀘스트를 수락하려면 다시 `퀘스트` 탭을 열고 `수락`을 눌러야 합니다.

개선안:
- 모달 취소는 이전 패널 상태를 보존하는 편이 자연스럽습니다.
- 의도적으로 닫는 정책이라면 취소 후 버튼 포커스나 시각 변화로 "패널로 돌아가지 않음"을 분명히 보여줘야 합니다.

### P2: 보상과 레벨업 정보가 체감되지 않음

관찰:
- 퀘스트 보상은 패널에 `GOLD 1 · EXP 10`으로 보였지만, 승리 후 실제 지급 내역은 로그에서 직접 보이지 않았습니다.
- 경험치는 시작부터 `100%`라 레벨업 가능 상태였고, 전투 보상 때문에 오른 것인지 알기 어렵습니다.
- 레벨업 후 상단은 `HP 20/35`, `MP 20/25`로 표시되지만 로그는 `HP 35 / MP 25`라고 적어 현재값과 최대값의 의미가 헷갈릴 수 있습니다.

개선안:
- 보상 지급 로그를 별도 카드로 보여주는 것이 좋습니다. 예: `보상: 금화 +1, 경험 +10`.
- 레벨업 로그는 `최대 HP 35, 최대 MP 25`처럼 최대치 증가임을 명확히 써야 합니다.
- 레벨업 가능 상태는 `경험 100%`만으로는 약하니 상단 또는 레벨업 버튼에 더 강한 상태 표시가 있으면 좋습니다.

### P2: 접근성 DOM에서 텍스트 중복이 많음

관찰:
- 주인공/NPC/퀘스트 패널에서 같은 값이 DOM 스냅샷에 두 번씩 노출됩니다. 예: `훈련 단검`, `금화(10) · 집중 부적`, 퀘스트 목표/요약 등이 반복됩니다.
- 시각적으로 큰 문제는 아니지만 스크린 리더와 자동화 테스트에서는 중복 읽힘 가능성이 높습니다.

개선안:
- 장식용/중복 텍스트에는 접근성 제외 처리를 검토해야 합니다.
- 정보 패널의 Row/Stat 계열 컴포넌트에서 값 텍스트가 두 경로로 렌더링되는지 확인하는 것이 좋습니다.

### P2: 새 게임 생성 화면의 시작 버튼 위치가 이른 편임

관찰:
- `시작` 버튼이 이름/성별/세계/종족/언어 선택보다 위에 있습니다.
- 기본값으로 바로 시작할 수 있다는 장점은 있지만, 처음 보는 사용자는 설정을 보기 전에 시작 버튼을 누를 수 있습니다.

개선안:
- 시작 버튼을 폼 하단으로 옮기거나, 상단 버튼은 유지하되 하단에도 같은 primary action을 제공하는 방식이 더 안전합니다.
- 선택 항목이 기본값임을 보여주는 요약이 있으면 실수 시작을 줄일 수 있습니다.

### P2: 영어 선택 시 클라이언트 로케일이 바뀌지 않음

관찰:
- 새 게임 화면에서 `English`를 눌러도 제목, 설명, 폼 placeholder, 버튼, 추천 행동은 모두 한국어입니다.
- 시작 후에도 상단 탭, 상태 라벨, 입력 placeholder, 추천 행동은 한국어이고 GM 첫 문장 일부만 영어입니다.

개선안:
- 클라이언트가 한국어만 지원한다면 언어 선택을 서버 서사 언어 선택으로 명명해야 합니다.
- 영어 UI를 지원하려면 `locale/en.ts`와 서버 display string의 영어 출력까지 같이 맞춰야 합니다.

### P3: 개발 콘솔 경고

관찰된 경고:
- `[Reanimated] Reduced motion setting is enabled`
- `props.pointerEvents is deprecated. Use style.pointerEvents`
- `"shadow*"` style props are deprecated. Use `boxShadow`.
- `Animated: useNativeDriver is not supported because the native animated module is missing`
- `You have set a custom wheel sensitivity...`

개선안:
- 릴리스 영향은 낮아 보이지만, web QA 노이즈를 줄이기 위해 `pointerEvents`, shadow 스타일, web animation 설정은 정리할 가치가 있습니다.

## 좋았던 점

- 모바일 한 화면에서 상태 바, 로그, 추천 행동, 입력창이 안정적으로 유지됩니다.
- 퀘스트 수락/공격 확인 모달은 명확하고 실수 방지에 도움이 됩니다.
- 전투 패널의 `공격 / 방어 / 도주 / 설득` 버튼 구조는 이해하기 쉽습니다.
- 레벨업 UI는 선택 전/후 상태와 확정 버튼 disabled 처리가 명확합니다.
- 새 이야기 시작은 확인 모달이 있어 현재 세션을 바로 잃지 않습니다.
- 소리 토글은 버튼 상태가 `소리 켜기` ↔ `소리 끄기`로 바뀌고, 클릭 시 추가 콘솔 에러는 보이지 않았습니다.
- 한국어 문체는 대체로 존댓말 톤을 유지합니다.
